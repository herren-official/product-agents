# Work: 구조화된 개발 워크플로

요구사항 파악 → 코드 리서치 → 계획 수립 → 검토 → 구현의 단계별 워크플로를 수행한다.

## 핵심 원칙

- **Phase 0~2에서는 코드 작성/수정 절대 금지**
- **Phase 간 전환은 체리님의 명시적 지시로만 진행**
- **각 Phase 완료 시 반드시 체리님에게 보고하고 다음 단계 승인 대기**

---

## Phase 0: 요구사항 파악

### 트리거
`/work` 실행 시 자동 시작

### 수행 내용
1. 체리님에게 **무엇을 만들/수정할 것인지** 질문
2. 다양한 입력 형태 수용:
   - Notion 링크 → `mcp__notionMCP__notion-fetch`로 카드 내용 파악
   - Figma URL → `mcp__Figma_Dev_Mode_MCP__get_figma_data`로 디자인 분석
   - 텍스트 설명 → 대화로 상세 요구사항 확인
   - 스크린샷 → 이미지 분석
3. 요구사항이 명확해질 때까지 대화로 보완
4. 최종 요구사항을 정리하여 체리님에게 확인

### 규칙
- ⛔ 코드 작성/수정 절대 금지
- ⛔ 코드 리서치도 아직 하지 않음
- ✅ 요구사항 파악에만 집중
- ✅ 모호한 부분은 반드시 질문

### 완료 조건
체리님이 요구사항을 확인/승인하면 Phase 1로 진행

---

## Phase 1: 코드 리서치

### 진입
체리님이 요구사항을 확인한 후 **플랜 모드로 진입**

```
EnterPlanMode → 다각도 코드 리서치 수행
```

### 수행 내용
요구사항과 관련된 코드를 **다각도로 디테일하게** 탐색:

1. **기존 패턴 분석**
   - 유사한 기능이 이미 구현되어 있는지 확인
   - 기존 구현 방식과 패턴 파악
   - 재사용 가능한 컴포넌트/유틸리티 탐색

2. **의존성 체인 추적**
   - API 레이어 (Service → Repository → Entity/Body)
   - ViewModel 레이어 (Contract → ViewModel)
   - UI 레이어 (Activity/Fragment → Screen → Component)
   - 데이터 흐름 전체 추적

3. **관련 파일 전수 조사**
   - 수정 대상 파일 목록
   - 영향받는 파일 목록
   - 참조해야 할 파일 목록

4. **컨벤션 확인**
   - `.docs/` 관련 문서 확인 (design-system, project-convention 등)
   - 기존 코드 스타일/패턴과의 일관성 확인

5. **Explore Agent 활용**
   - 복잡한 탐색은 `Agent(subagent_type=Explore)` 병렬 활용
   - 여러 방향의 리서치를 동시에 수행

### 보고 형식
리서치 완료 후 체리님에게 다음을 보고:
- 관련 파일 목록 (경로 + 역할)
- 기존 패턴/유사 구현 사례
- 주의사항 및 제약사항
- 구현 시 고려해야 할 포인트

### 규칙
- ⛔ 코드 작성/수정 절대 금지 (플랜 모드)
- ✅ Read, Glob, Grep, Agent(Explore) 활용
- ✅ 가능한 한 넓고 깊게 탐색
- ✅ 발견한 내용을 체계적으로 보고

### 완료 조건
리서치 보고 후 체리님이 Phase 2 진행을 지시

---

## Phase 2: 구현 계획 수립

### 진입
플랜 모드 유지 상태에서 수행

### 수행 내용
1. Phase 1 리서치 결과를 기반으로 **구현 계획 작성**
2. 계획을 `.docs/local/plans/` 에 `.md` 파일로 저장

### 계획 파일 구조
```markdown
# {GBIZ-번호}: {작업 제목}

## 요구사항
- (Phase 0에서 파악한 내용)

## 리서치 요약
- (Phase 1에서 파악한 핵심 사항)

## 수정 대상 파일
| 파일 | 변경 내용 | 비고 |
|------|----------|------|
| path/to/file.kt | 변경 설명 | |

## 구현 순서
### Step 1: {제목}
- 파일: `path/to/file.kt`
- 변경: 구체적인 변경 내용

### Step 2: {제목}
- ...

## 주의사항
- (컨벤션, 기존 패턴과의 일관성, 잠재적 이슈 등)
```

3. 계획 파일 저장 후 **플랜 모드 퇴장**
4. 체리님에게 계획 파일 경로 안내

### 규칙
- ⛔ 코드 작성/수정 절대 금지
- ✅ 계획은 최대한 구체적으로 (어떤 파일의 어떤 부분을 어떻게 수정)
- ✅ 구현 순서는 의존성을 고려하여 결정
- ✅ 기존 컨벤션/패턴을 준수하는 방향으로 계획

### 완료 조건
계획 파일 저장 + 플랜 모드 퇴장 + 체리님에게 안내

---

## Phase 3: 검토 루프

### 수행 내용
1. 체리님이 `.docs/local/plans/*.md` 파일을 직접 검토
2. 피드백 반영:
   - **수정사항 있음** → 플랜 모드 재진입 → Phase 1 (추가 리서치) 또는 Phase 2 (계획 수정) 반복
   - **승인** → Phase 4로 진행

### 규칙
- ⛔ 체리님 승인 없이 Phase 4 진행 금지
- ✅ 수정 요청 시 해당 Phase로 정확히 돌아가기
- ✅ 계획 파일도 수정사항 반영하여 업데이트

### 완료 조건
체리님이 "진행해", "구현해", "ㄱㄱ" 등 승인 의사 표시

---

## Phase 4: 구현

### 진입
체리님 승인 후 일반 모드에서 수행

### 수행 내용
1. `.docs/local/plans/*.md` 파일의 계획을 다시 읽기
2. 계획에 명시된 **구현 순서대로** 코드 작성
3. 각 Step 완료 시 진행 상황 간략 보고
4. 전체 구현 완료 후 빌드 확인 (`./gradlew compileDevDebugKotlin`)

### 규칙
- ✅ 계획에 명시된 내용만 구현 (범위 초과 금지)
- ✅ 기존 컨벤션/패턴 준수
- ✅ 빌드 성공 확인 필수
- ✅ 구현 완료 후 체리님에게 결과 보고

---
