# reviewer

변경된 코드를 코드 리뷰해준다.

## 입력 모드

- **인자가 있을 때** (`$ARGUMENTS`): GitHub PR URL로 판단하고, PR의 변경 파일을 리뷰한다.
- **인자가 없을 때**: 현재 브랜치에서 작업 중인 변경사항을 리뷰한다.

## 절차

### 모드 1: PR URL이 주어진 경우

1. PR URL에서 PR 번호를 추출한다.
2. `gh pr view {pr_number} --json baseRefName,headRefName,files,title,body`로 PR 정보를 가져온다.
3. `gh pr diff {pr_number}`로 변경된 diff를 가져온다.
4. diff에서 변경된 파일 목록을 파악하고, 각 파일의 전체 내용을 Read 도구로 읽는다.
   - 해당 브랜치의 파일을 읽기 위해 필요 시 `git show {headRef}:{file_path}`를 사용한다.

### 모드 2: 인자가 없는 경우 (현재 작업 코드 리뷰)

1. `git branch --show-current`로 현재 브랜치명을 확인한다.
2. 현재 브랜치의 base 브랜치를 추정한다:
   - `gh pr view --json baseRefName`으로 이미 PR이 있으면 그 base를 사용한다.
   - PR이 없으면 `git merge-base HEAD develop`으로 develop 브랜치와의 분기점을 찾는다.
3. `git diff {base}...HEAD --name-only`로 변경된 파일 목록을 가져온다.
4. `git diff {base}...HEAD`로 diff를 가져온다.
5. 변경된 파일의 전체 내용을 Read 도구로 읽는다.
6. 스테이징되지 않은 변경사항도 있으면 `git diff`로 추가 확인한다.

### 공통 절차

5. 프로젝트 컨벤션 문서들을 읽어서 리뷰 기준을 파악한다:
   - `.docs/conventions/code-convention.md`
   - `.docs/conventions/architecture-improvement-proposal.md`
   - `CLAUDE.md`
6. **비즈니스 로직 파악**: 변경된 파일뿐 아니라, 관련된 도메인 객체·서비스·인터페이스도 읽어서 전체 맥락을 파악한다.
   - 변경된 클래스가 의존하는 클래스, 호출하는 메서드의 실제 구현을 확인한다.
   - 도메인 용어와 비즈니스 규칙을 이해한 뒤 리뷰한다.
   - 코드 구조에 대한 지적은 비즈니스 로직상 타당한 이유가 있는지 먼저 확인한다.
7. 변경된 파일들을 아래 리뷰 기준에 따라 분석한다.

## 리뷰 기준 (우선순위 순)

### 1. 성능 이슈 (최우선)
- N+1 쿼리 문제
- 불필요한 DB 조회 (루프 안에서 쿼리, 사용하지 않는 fetch join 등)
- 대량 데이터 처리 시 메모리 이슈 (전체 로딩 vs 페이징/스트리밍)
- 불필요한 객체 생성, 비효율적 컬렉션 연산
- 캐싱 누락 또는 잘못된 캐시 전략
- 트랜잭션 범위가 불필요하게 넓은 경우 (특히 외부 API 호출 포함 여부)
- 인덱스 미활용 가능성이 있는 쿼리
- 벌크 연산 가능한 곳에서 건별 처리 (`save()` 루프 vs `saveAll()`)
- 대량 IN 절 쿼리의 파티셔닝 필요 여부

### 2. 코드 컨벤션 준수
- `.docs/conventions/code-convention.md` 기준:
  - 로직 단순화 원칙 (중복 코드, 조건문, 불필요한 코드)
  - 아키텍처 개선 원칙 (레이어 분리, 의존성 방향)
  - 예외 처리 원칙
  - 가독성 향상 원칙
  - Facade 패턴 사용 가이드
- CLAUDE.md 기준:
  - 모듈별 Spring Boot 버전에 맞는 패키지 사용 (javax vs jakarta)
  - 레이어별 책임 분리

### 3. 객체간 책임/역할/협력
- 단일 책임 원칙(SRP) 위반 여부
- 서비스 레이어에 도메인 로직이 과도하게 존재하는지
- 도메인 객체가 빈약한 모델(Anemic Domain Model)이 아닌지
- 의존성 방향이 올바른지 (상위 레이어 -> 하위 레이어)
- Facade가 필요한 상황에서 Service가 다른 Service를 직접 호출하는지
- Repository에 비즈니스 로직이 포함되어 있는지
- ⚠️ 코드 구조 지적 시, 비즈니스 로직상 현재 구조가 타당한 이유가 있는지 반드시 확인한다.

### 3-1. 데이터 소스 일관성
- 같은 도메인/개념의 기존 쿼리들과 **동일한 데이터 소스(테이블/엔드포인트)**를 쓰는가?
- 이번 변경만 다른 테이블을 쓴다면 근거가 PR 본문 또는 태스크 문서에 있는가?
- 대상 테이블이 **append-only**(예: `*_history`)인 경우, 수정/삭제 후 과거 row 잔존으로 인한 false positive 가능성은 검토됐는가?
- 같은 파일/클래스 안에 다른 테이블을 쓰는 유사 쿼리가 공존한다면 비대칭 사유를 명시적으로 지적한다.

### 4. 일반 코드 품질
- 버그 가능성 (NPE, 경계값, 동시성 이슈)
- 보안 취약점 (SQL Injection, XSS 등)
- 에러 핸들링 누락
- 미사용 코드 (dead code) 잔존 여부
- 네이밍 적절성
- 하드코딩된 문자열의 상수 추출 필요 여부

## 출력 형식

```
## PR 리뷰: [PR 제목] (#번호)
- **base**: {base_branch} <- **head**: {head_branch}
- **변경 파일**: {파일 수}개

---

### 🔴 Critical (반드시 수정 필요)
> 성능 이슈, 버그, 보안 취약점

#### 1. [파일명:라인] 이슈 제목
- **카테고리**: 성능 / 버그 / 보안
- **문제**: 구체적인 문제 설명
- **제안**: 개선 방안 (코드 예시 포함)

---

### 🟡 Warning (개선 권장)
> 컨벤션 위반, 책임 분리 이슈

#### 1. [파일명:라인] 이슈 제목
- **카테고리**: 컨벤션 / 아키텍처 / 책임분리
- **문제**: 구체적인 문제 설명
- **제안**: 개선 방안

---

### 🟢 Suggestion (선택적 개선)
> 가독성, 네이밍, 코드 스타일

#### 1. [파일명:라인] 이슈 제목
- **제안**: 개선 방안

---

### 👍 잘된 점
- 잘 작성된 부분이 있으면 언급

### 📊 요약
- Critical: N건 / Warning: N건 / Suggestion: N건
```

## 규칙

- 한국어로 리뷰를 작성한다.
- **비즈니스 로직을 먼저 파악한 뒤 리뷰한다.** 코드 구조를 지적하기 전에, 해당 구조가 비즈니스 요구사항 때문에 그렇게 된 것인지 확인한다.
- 추측이 아닌 코드를 읽고 근거를 기반으로 리뷰한다.
- 변경된 파일만 리뷰하되, 관련 파일(의존 클래스, 호출 대상 등)의 맥락도 반드시 파악한다.
- 사소한 스타일 지적보다 실질적인 문제에 집중한다.
- 변경하지 않은 기존 코드에 대해서는 리뷰하지 않는다.
- diff에서 추가/수정된 라인 위주로 리뷰한다.
- 리뷰 지적에 대해 반론이 오면, 비즈니스 로직을 다시 확인하고 틀렸으면 솔직하게 인정하되 코드 품질 문제가 여전히 유효하면 근거와 함께 유지한다.
