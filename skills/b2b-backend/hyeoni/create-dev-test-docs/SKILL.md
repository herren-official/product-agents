# 개발자 테스트 케이스 문서 생성

다음 정보를 바탕으로 개발자 테스트 케이스 문서를 자동 생성해주세요.

## 필요한 정보

### 사용자에게 질문할 정보

1. **문서 저장 경로** (필수): 문서를 저장할 디렉토리 경로 (예: `.claude/docs/cosmetic_statistics/`)
2. **API 엔드포인트** (필수): 테스트할 API 엔드포인트 (예: `GET /api/v2/{shopNo}/statistics/cosmetics`)
3. **PRD 문서 경로** (선택): PRD 문서 파일 경로
   - 제공되면 해당 파일을 읽어서 PRD 내용 참조
   - 제공되지 않으면 코드 분석만으로 문서 생성
4. **DB 환경** (선택): `dev` 또는 `prod` (기본값: `dev`)
   - gongbiz-db skill을 사용하여 실제 데이터 조회

### 코드와 PRD 분석으로 자동 추출할 정보

1. **기능 이름**: Controller와 Service 코드에서 추출
2. **테스트 목적**: API 기능과 비즈니스 로직을 분석하여 자동 생성
3. **포함 범위**: 구현된 기능과 Service 메서드들을 분석하여 자동 생성
4. **제외 범위**: 구현되지 않은 관련 기능들을 분석하여 자동 생성

## 테스트 케이스 작성 가이드

### 필수 카테고리 (기능에 따라 선택)

1. **권한 테스트** (권한이 있는 기능인 경우)
   - 원장/관리자 권한 테스트
   - 직원 권한 테스트
   - 권한 없는 사용자 테스트

2. **조회 및 필터링** (조회 기능인 경우)
   - 데이터 있음/없음
   - 기간별 필터링
   - 담당자별 필터링
   - 상태별 필터링

3. **CRUD 테스트**: 정상 등록/수정/삭제, 중복 등록 방지

4. **정렬 및 페이징**: 정렬 기준, 페이징 처리, 빈 페이지

5. **데이터 정합성**: 데이터 형식, 계산 로직, 필수 필드, Null 처리

6. **예외 처리**: 필수 파라미터 누락, 잘못된 데이터 형식, 범위 초과, 중복 방지

7. **회귀 테스트**: 기존 기능 동작 확인, 연관 기능 동작 확인

## 작성 시 주의사항

1. **간결하게 작성**: 핵심 테스트 케이스만 포함 (각 카테고리당 2-5개)
2. **명확한 조건**: 테스트 조건을 명확하게 기재
3. **측정 가능한 결과**: 기대 결과는 명확하게 측정/확인 가능하도록 작성
4. **실제 구현 기반**: 실제로 구현한 기능과 코드를 바탕으로 작성
5. **Backend 중심**: Backend 개발자이므로 API, 데이터 로직, DB 쿼리 중심으로 작성
6. **자명한 내용 제외**: "HTTP 200 응답", "정상 동작" 같은 너무 당연한 기대 결과는 제외
7. **DB 검증 필수**: 조건/기대 결과는 코드 추정이 아닌 dev DB 실제 데이터 조회 기반으로 구체적 수치 기재
8. **응답 기준 작성**: API 응답에 포함되지 않는 내부 계산용 값은 기대 결과에서 제외
9. **노션 호환**: 테이블 셀 내 항목 구분은 ` - ` 사용. `<br>`, `•` 등 HTML 태그나 특수문자 사용 금지
10. **계산값 검증**: 객단가, 비율 등 계산 필드는 Calculator 코드의 반올림 모드(scale, RoundingMode)를 확인한 후 정확한 계산값 기재

## 실행 단계

1. **사용자에게 정보 질문** (필수: 저장 경로, API 엔드포인트 / 선택: PRD 경로, DB 환경)

2. **코드 및 PRD 분석**
   - API 엔드포인트로 Controller 파일 찾기
   - Controller에서 Service 메서드 및 비즈니스 로직 분석
   - 관련 테스트 코드 찾아서 분석
   - PRD 문서 가져오기 (제공된 경우)
   - 기능 이름, 테스트 목적, 포함/제외 범위 자동 추출

3. **DB 데이터 조회 (gongbiz-db skill 활용)**
   - 테이블 구조 파악: Entity 코드에서 테이블명, 컬럼명, 타입 확인
   - Infrastructure 레이어 분석: Repository/Reader 코드에서 실제 쿼리 조건 파악
   - dev DB 조회: 테스트 기간 내 실제 데이터 조회
   - 계산 로직 검증: Calculator, Util 코드의 반올림 모드 확인
   - 응답 DTO 확인: 실제 API 응답에 포함되는 필드만 기대 결과에 기재

4. **테스트 케이스 작성**
   - 기능의 특성 파악 (조회/CRUD/통계 등)
   - 적합한 테스트 케이스 카테고리 선택
   - 각 카테고리별로 2-5개의 핵심 테스트 케이스 작성
   - DB 조회 결과 기반으로 구체적인 조건/기대 결과 작성

5. **문서 생성**
   - 지정된 경로에 `[{기능 이름}] 개발자 테스트 케이스.md` 파일로 저장
   - 일정, Figma 링크, 정책 문서 링크는 빈 칸으로 남겨두기

6. **결과 출력**
   - 생성된 문서의 경로 출력
   - 작성된 테스트 케이스 개수 요약
