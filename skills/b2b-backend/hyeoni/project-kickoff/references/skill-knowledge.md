## 프로젝트 킥오프 전문 지식

### 수집 원칙

1. **필수 vs 선택 분리**: 노션/피그마/에픽번호는 필수. DB/AWS/Slack/유사 기능은 선택. 선택 항목이 없어도 수집을 진행할 수 있어야 한다.
2. **병렬 우선**: Phase 1의 수집 단계들은 서로 독립적이다. 항상 병렬 실행한다.
3. **부분 실패 허용**: 한 소스 실패가 전체 수집을 막지 않는다. 실패한 소스는 리포트에 "수집 실패" 섹션으로 남긴다.
4. **출처 명시**: 모든 요약 항목에는 원문 URL/파일 경로를 명시한다. 추후 검증 가능해야 한다.
5. **중복 제거**: 여러 소스에서 같은 정보가 나오면 하나로 합치되, 출처는 모두 나열한다.
6. **사실 vs 추론 구분**: 문서에 적힌 사실은 그대로 인용, 유추한 내용은 "(추론)" 라벨을 붙인다.

### 출력 파일 구조

`.claude/docs/{epic-project-name}/` 하위에 아래 파일들을 생성한다. (수집 생략된 단계는 해당 파일을 만들지 않거나 "스킵됨" 1줄만 남긴다)

| 파일 | 용도 | 소스 | 크기 가이드 |
|---|---|---|---|
| `context.md` | 얇은 개요 + 카테고리별 파일로의 네비게이션 허브 | `overview-template.md` | 150~300줄 |
| `design-draft.md` | **설계 문서 초안** — AS-IS/TO-BE, DDL, API, Phase 분할, 설계 결정. 엔지니어가 이어받음 | `design-draft-template.md` | 400~800줄 |
| `policy-summary.md` | 노션 정책 요약 (원문 인용 + 바로가기) | — | 100~300줄 |
| `design-notes.md` | 피그마 분석 | — | 50~200줄 |
| `db-schema.md` | DB 스키마 | — | 테이블 수에 비례 |
| `codebase-map.md` | 영향받는 코드 위치 | — | 100~300줄 |
| `infra-map.md` | AWS 인프라 | — | 50~150줄 |
| `slack-notes.md` | Slack 의사결정 (선택) | — | 30~100줄 |
| `agent-team.md` | 에이전트 팀 제안 | `agent-team-guide.md` | 80~150줄 |

**context.md와 design-draft.md의 역할 분리**
- `context.md`는 **훑어보기용**. 외부 공유·팀 킥오프 미팅에서 "어떤 프로젝트인가"를 5분 안에 파악하게 하는 목적.
- `design-draft.md`는 **설계 문서 초안**. 엔지니어가 채워넣을 슬롯이 명시돼 있고, **[자동] / [자동·검증 필요] / [수동 검토 필요]** 라벨로 자동 생성 범위를 표시한다. 노션에 업로드될 최종 설계 문서의 시작점이다.

### 요약 품질 기준

- 각 소스를 **한 번씩만 읽고** 요약한다. 같은 링크를 여러 번 fetch하지 않는다.
- 코드/스키마/인프라는 **식별자(경로/이름)를 정확히 인용**한다. "대충 이 근처" 같은 모호한 참조 금지.
- 리스크 섹션은 반드시 포함한다. "모든 게 명확하다"고 느껴질 때도 불확실성 1개 이상은 찾아 기록한다.
- 다음 단계 제안은 **실행 가능한 커맨드**로 제시한다. 예: `/plan`, `/api-test-plan`, 특정 에이전트 prompt 예시.

### design-draft.md 작성 품질 기준 (필수)

1. **라벨 의무**: 모든 섹션 제목 옆에 `[자동]` / `[자동·검증 필요]` / `[수동 검토 필요]` 중 하나를 붙인다. 엔지니어가 "이 부분은 자동 추출인가, 판단이 필요한가"를 즉시 알 수 있어야 한다.
2. **Mermaid 시퀀스**: AS-IS와 TO-BE 각각 최소 하나씩. 엔드포인트가 확정되지 않았어도 스켈레톤은 반드시 들어간다 (2.2 섹션).
3. **설계 결정 포맷**: 7절의 각 결정은 반드시 `결정 / 근거 / 트레이드오프 / **결론**` 4요소를 갖춘다. "결론" 라벨은 굵은 글씨로.
4. **Phase 표 필수**: 1.4에 Phase 표(Phase/시점/작업범위) + Phase별 작업 상세 bullet. 최소 Phase 1은 채운다.
5. **영향 범위 표 의무**: 8절 표의 각 행은 `파일경로:라인번호` 또는 `파일경로` 인용을 포함. 인용 없이 "어디선가 수정" 금지.
6. **DDL 블록**: 신규 테이블이 있으면 4.1에 완전한 `CREATE TABLE` SQL 블록을 넣는다. 컬럼 타입/제약/COMMENT 모두 포함. 확정 안 된 부분은 `{TODO: 타입 확정}` 형태로 명시.
7. **롤백 체크리스트**: 4.3.1에 최소 3개 체크박스. "롤백 방법 없음"으로 넘기지 말고 "INSERT only라 롤백 불필요" 같은 근거를 남긴다.
8. **논의 필요 항목**: 10절 표에 최소 1개 항목. 없어 보일 때도 "노션/피그마 원문에서 명시되지 않은 부분"을 찾아 등록.
9. **출처 섹션 하단 필수**: 모든 자동 추출 정보의 원문 링크를 최하단 "출처"에 모아둔다.

### 추론 표시 규칙

노션/피그마에 **명시되지 않은** 내용을 추론으로 작성할 때는 문장 끝에 `(추론)` 라벨을 붙인다.
예: "마이그레이션은 사전 실행이 안전하다 (추론)"
엔지니어가 "이건 검증이 필요하겠구나"를 바로 인지할 수 있어야 한다.

### 에이전트 팀 제안 원칙

1. **필요한 에이전트만**: 도메인에 관련된 에이전트만 추천한다. 모든 에이전트를 나열하지 않는다.
2. **호출 시점 명시**: 각 에이전트마다 "언제 호출하는지"를 구체적으로 적는다 (예: "DB 스키마 변경이 확정된 후", "구현 완료 후 QA 전에").
3. **호출 예시 prompt 포함**: 바로 복사해서 쓸 수 있는 1~2줄 prompt 예시를 함께 제공한다.
4. **병렬화 포인트 표시**: 동시에 돌려도 되는 조합을 별도로 표시한다.
