대상 코드에 대해 테스트 작성, 실행, 디버깅, 리뷰를 자동으로 수행해라. 모든 테스트가 통과하고 품질 기준을 충족할 때까지 반복한다.

> 테스트 케이스 도출은 이 커맨드의 범위가 아니다. 케이스를 먼저 파악하려면 `/test [대상]`을 실행한다.
> `/test`가 생성한 테스트 계획 파일이 있으면 이를 참조하여 작성한다.

## 입력

$ARGUMENTS

- 인자가 파일 경로이면 해당 파일을 대상으로 한다.
- 인자가 클래스명이면 프로젝트에서 해당 클래스를 검색하여 대상으로 한다.
- 인자가 없으면 현재 브랜치와 develop 브랜치를 비교(`git diff develop...HEAD --name-only`)하여 추가/변경된 프로덕션 코드 파일(.java, .kt)을 대상으로 한다. 테스트 파일(`src/test/`)은 대상에서 제외한다. develop 브랜치가 없으면 main, master 순으로 시도한다.

## 절차

### Phase 1: 환경 파악
다음을 한 번만 확인한다:
- 빌드 도구: `build.gradle.kts` / `build.gradle` / `pom.xml`
- 언어: Java / Kotlin (대상 파일 확장자로 판단)
- Spring Boot 버전: 2.x → `javax.*`, 3.x → `jakarta.*`
- 테스트 라이브러리: MockK / Mockito / AssertJ / JUnit5
- 기존 테스트 컨벤션 (네이밍, fixture 방식, assertion 스타일)

### Phase 2: 테스트 계획 파일 확인 및 테스트 작성

#### Step 2-0: 테스트 계획 파일 확인
- 현재 브랜치명을 `git branch --show-current`로 가져온다.
- 브랜치명의 `/`를 `-`로 치환한다.
- 프로젝트 루트의 `.claude/test/{branch-name}-test-plan.md` 파일이 존재하는지 확인한다.
- **파일이 있으면**: 해당 계획의 케이스 목록을 기준으로 테스트를 작성한다.
- **파일이 없으면**: 대상 파일을 직접 분석하여 케이스를 도출하고 작성한다 (기존 동작).

#### Step 2-1: 테스트 코드 작성
`/test-write` 커맨드와 동일한 절차로 테스트 코드를 작성한다:
- 기존 테스트 파일이 있으면 읽어 이미 작성된 케이스를 파악하고, 누락된 케이스만 추가한다.
- 없으면 새 테스트 파일을 생성한다.
- 프로젝트의 기존 테스트 컨벤션을 따른다.

### Phase 3: 반복 실행 (최대 3회)

아래 사이클을 최대 3회 반복한다. 반복 횟수를 `[N/3]` 형식으로 표시한다.

#### Step 3-1: 테스트 실행
- Gradle: `./gradlew test --tests "[패키지].[테스트클래스명]"` 실행.
- Maven: `./mvnw test -Dtest=[테스트클래스명]` 실행.
- `gradlew`에 실행 권한이 없으면 `chmod +x ./gradlew` 후 재시도한다.
- 실행 시간이 5분을 초과하면 중단하고 "타임아웃"으로 표시한다.

#### Step 3-2: 결과 판정
- **모두 성공** → Step 3-3(리뷰)로 진행한다.
- **실패 또는 컴파일 에러 존재** → Step 3-2a(디버깅)로 진행한다.

#### Step 3-2a: 디버깅 및 수정
`/test-debug` 커맨드와 동일한 절차로 실패를 분석한다:
- 실패 원인을 분류한다: 컴파일 에러 / 테스트 코드 결함 / 프로덕션 코드 회귀 / 환경 문제 / Flaky 테스트.
- **테스트 코드 결함 또는 컴파일 에러**: 테스트 코드를 직접 수정한다.
- **프로덕션 코드 회귀**: 프로덕션 코드를 수정하지 않는다. 해당 테스트를 `@Disabled("프로덕션 코드 수정 필요: [사유]")`로 표시하고 Phase 4 결과에 기록한다.
- **환경 문제**: 설정 수정을 시도한다. 해결 불가 시 해당 테스트를 `@Disabled("환경 설정 필요: [사유]")`로 표시하고 Phase 4 결과에 기록한다.
- 수정 후 Step 3-1로 돌아간다.

#### Step 3-3: 테스트 리뷰
`/test-review` 커맨드와 동일한 기준으로 품질을 평가한다:
- 커버리지 누락이 있는가?
- CRITICAL 품질 문제(안티패턴)가 있는가?

판정:
- **CRITICAL 없음 + 커버리지 누락 없음** → Phase 4(완료)로 진행한다.
- **CRITICAL 존재 또는 커버리지 누락** → 테스트 코드를 수정/추가하고 Step 3-1로 돌아간다.

### Phase 4: 최종 결과 출력

## 출력 형식

```markdown
## 테스트 자동화 결과

### 대상
- 프로덕션 코드: `[경로]` ([N]개 파일)
- 언어/프레임워크: [Java|Kotlin] / Spring Boot [버전]

### 실행 이력
| 회차 | 실행 결과 | 성공 | 실패 | 수정 내용 |
|------|----------|------|------|-----------|
| 1/3 | 실패 | 8 | 2 | Mock 설정 오류 수정, import 추가 |
| 2/3 | 실패 | 9 | 1 | assertion 기대값 수정 |
| 3/3 | 성공 | 10 | 0 | — |

### 최종 테스트 현황
- 생성/수정 파일: `[경로]`
- 전체 테스트: [N]개 (Unit [N] / Slice [N] / Integration [N])
- 성공: [N]개 / 실패: [N]개 / @Disabled: [N]개

### @Disabled 테스트 (수동 조치 필요)
| 테스트 | 사유 | 필요 조치 |
|--------|------|-----------|
| should_fail_when_invalid | 프로덕션 코드 회귀 | OrderService.validate() 로직 확인 필요 |

### 품질 리뷰 요약
- CRITICAL: [N]개 / WARNING: [N]개 / INFO: [N]개
- 커버리지 누락: [N]개 케이스

### 전체 판정
- [완료: 모든 테스트 통과 + 품질 기준 충족]
- [부분 완료: 테스트 통과하나 @Disabled [N]개 존재]
- [미완료: 3회 반복 후에도 실패 존재 — 수동 개입 필요]

### 다음 단계
- @Disabled 테스트가 있으면 프로덕션 코드를 수정한 후 `/test-debug [테스트 클래스]`로 재확인하세요.
- 프로덕션 코드 변경의 영향 범위를 확인하려면 `/test-impact [파일]`를 실행하세요.
- 전체 코드 리뷰가 필요하면 `/review`를 실행하세요.
```

## 제약

- 프로덕션 코드를 수정하지 않는다. 프로덕션 코드 변경이 필요한 경우 `@Disabled`로 표시하고 사유를 기록한다.
- 반복 횟수는 최대 3회로 제한한다. 3회 후에도 실패가 남으면 "미완료"로 판정하고 수동 조치가 필요한 항목을 명시한다.
- 프로젝트의 기존 테스트 컨벤션을 최우선으로 따른다.
- 대상 파일을 찾을 수 없으면 "대상 파일을 찾을 수 없습니다. 파일 경로 또는 클래스명을 지정해 주세요."를 출력한다.
- develop/main/master 브랜치를 모두 찾을 수 없으면 "기준 브랜치를 찾을 수 없습니다. 대상 파일을 직접 지정해 주세요."를 출력한다.
- 빌드 도구나 Spring Boot 버전을 확인할 수 없으면 사용자에게 확인을 요청한다.
