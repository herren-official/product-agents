실패한 테스트의 원인을 분석하고 수정 방향을 제시해라.

## 입력

$ARGUMENTS

- 인자가 테스트 클래스명 또는 메서드명이면 해당 테스트를 대상으로 한다.
- 인자가 파일 경로이면 해당 테스트 파일을 대상으로 한다.
- 인자가 CI 로그 텍스트(또는 URL)이면 로그에서 실패한 테스트를 추출하여 대상으로 한다.
- 인자가 없으면 현재 브랜치와 develop 브랜치를 비교(`git diff develop...HEAD --name-only`)하여 변경된 테스트 파일을 대상으로 한다. develop 브랜치가 없으면 main, master 순으로 시도한다.

## 절차

### Step 1: 실패 재현
- 빌드 도구 판단: `gradlew` 파일 존재 시 Gradle, `mvnw` 또는 `pom.xml`만 있으면 Maven.
- Gradle: `./gradlew test --tests "[테스트클래스명]"` 실행.
- Maven: `./mvnw test -Dtest=[테스트클래스명]` 실행.
- 실행 결과(성공/실패/컴파일에러)를 기록한다.
- `gradlew`에 실행 권한이 없으면 `chmod +x ./gradlew` 후 재시도한다.

### Step 2: 실패 원인 분류
실패를 다음 유형으로 분류한다:

| 유형 | 판단 기준 |
|---|---|
| **컴파일 에러** | import 누락, 존재하지 않는 메서드/클래스 참조, 어노테이션 오류 |
| **테스트 코드 결함** | assertion 실패, Mock 설정 오류, 잘못된 기대값 |
| **프로덕션 코드 회귀** | 프로덕션 코드 변경으로 기존 테스트가 깨짐 |
| **환경 문제** | DB 연결 실패, Testcontainers 실행 불가, 포트 충돌 |
| **Flaky 테스트** | 동일 코드에서 성공/실패가 반복됨 |

### Step 3: Flaky 테스트 탐지 (해당 시)
다음 패턴이 있으면 Flaky 테스트로 의심한다:
- `Thread.sleep()` 사용
- `LocalDateTime.now()` 또는 시간 의존 로직
- 랜덤 데이터 사용 (`Random`, `UUID.randomUUID()` 등)
- 테스트 실행 순서에 의존하는 공유 상태
- `@SpringBootTest(webEnvironment = DEFINED_PORT)` (포트 충돌)

### Step 4: 원인 분석
- 테스트 코드와 프로덕션 코드를 읽는다.
- 실패 스택트레이스를 분석한다.
- 최근 변경 이력(`git log --oneline -10`)을 확인하여 변경과 실패의 연관성을 파악한다.
- 환경 문제인 경우 설정 파일(application-test.yml 등)을 확인한다.

### Step 5: 수정 방향 제시
- 원인 유형별로 구체적 수정 방향을 제시한다.
- 프로덕션 코드 회귀인 경우, 테스트를 수정해야 하는지 프로덕션 코드를 수정해야 하는지 판단한다.
- Flaky 테스트인 경우, 안정적인 대안 코드를 제시한다.

## 출력 형식

```markdown
## 테스트 실패 분석

### 대상
- 테스트: `[클래스명#메서드명]`
- 프로덕션 코드: `[경로]`
- 실행 결과: [실패 / 컴파일에러 / 성공(재현 불가)]

### 원인 분류
**[컴파일 에러 / 테스트 코드 결함 / 프로덕션 코드 회귀 / 환경 문제 / Flaky 테스트]**

### 원인 상세
[스택트레이스 핵심 부분과 원인 설명]

### 수정 방향
| 수정 대상 | 파일 | 변경 내용 |
|---|---|---|
| 테스트 코드 | `[경로]` | [구체적 수정 내용] |
| 프로덕션 코드 | `[경로]` | [구체적 수정 내용] (해당 시) |
| 설정 | `[경로]` | [구체적 수정 내용] (해당 시) |

### 수정 코드
\`\`\`[java|kotlin]
// 수정 전
...
// 수정 후
...
\`\`\`

### 다음 단계
- 수정 후 테스트를 재실행하여 확인하세요.
- 테스트 품질을 검증하려면 `/test-review [테스트 클래스]`를 실행하세요.
```

## 제약

- 실패 원인이 불분명하면 가능한 원인을 2~3가지 나열하고, 각각의 확인 방법을 제시한다.
- 프로덕션 코드 수정이 필요한 경우, 테스트를 우회하는 방식이 아니라 근본 원인을 수정하는 방향을 제안한다.
- 대상 파일을 찾을 수 없으면 "대상 테스트 파일을 찾을 수 없습니다. 테스트 클래스명 또는 파일 경로를 지정해 주세요."를 출력한다.
- develop/main/master 브랜치를 모두 찾을 수 없으면 "기준 브랜치를 찾을 수 없습니다. 대상 파일을 직접 지정해 주세요."를 출력한다.
