프로덕션 코드 변경에 영향받는 테스트를 분석하고, 커버리지 갭을 식별해라.

## 입력

$ARGUMENTS

- 인자가 파일 경로이면 해당 프로덕션 코드 파일의 변경 영향을 분석한다.
- 인자가 클래스명이면 프로젝트에서 해당 클래스를 검색하여 대상으로 한다.
- 인자가 없으면 현재 브랜치와 develop 브랜치를 비교(`git diff develop...HEAD --name-only`)하여 추가/변경된 프로덕션 코드 파일(.java, .kt)을 대상으로 한다. 테스트 파일(`src/test/`)은 대상에서 제외한다. develop 브랜치가 없으면 main, master 순으로 시도한다.

## 절차

### Step 1: 변경 범위 파악
- 대상 프로덕션 코드를 읽는다.
- `git diff develop...HEAD -- [파일경로]`로 변경된 메서드/필드/시그니처를 식별한다.
- 변경 유형을 분류한다:
  - **시그니처 변경**: 메서드명, 파라미터, 반환 타입 변경 → 기존 테스트 컴파일 에러 가능
  - **로직 변경**: 분기 조건, 계산 로직, 예외 처리 변경 → 기존 테스트 assertion 실패 가능
  - **의존성 변경**: 새 의존성 추가, 기존 의존성 제거 → Mock 설정 변경 필요
  - **신규 추가**: 새 메서드/클래스 추가 → 신규 테스트 필요

### Step 2: 영향받는 테스트 검색
- `src/test/` 하위에서 변경된 클래스를 import하거나 참조하는 테스트 파일을 검색한다.
- 변경된 메서드를 직접 호출하는 테스트 메서드를 식별한다.
- 간접 영향(변경 클래스를 의존하는 다른 클래스의 테스트)도 1단계까지 추적한다.

### Step 3: 영향 분석
각 영향받는 테스트에 대해 다음을 판단한다:
- **깨질 가능성**: 시그니처 변경이면 높음, 로직 변경이면 중간, 신규 추가이면 없음
- **수정 필요 여부**: Mock 재설정, assertion 값 변경, 새 테스트 메서드 추가 등
- **수정 난이도**: 단순(값 변경) / 보통(Mock 재구성) / 복잡(테스트 구조 변경)

### Step 4: 커버리지 갭 분석
- 변경된 메서드/로직 중 테스트가 존재하지 않는 항목을 식별한다.
- 변경된 분기 조건 중 테스트되지 않은 경로를 식별한다.
- 신규 추가된 메서드 중 테스트가 없는 항목을 식별한다.

## 심각도 분류
- 🔴 필수 수정 (CRITICAL): 장애, 보안 취약점, 데이터 손실 가능성
- 🟡 권장 수정 (WARNING): 성능 저하, 유지보수 비용 증가, 안티패턴
- 🟢 참고 (INFO): 더 나은 대안, 스타일, 개선 아이디어

## 출력 형식

```markdown
## 변경 영향 분석

### 변경 대상
| 파일 | 변경 유형 | 변경 요약 |
|---|---|---|
| `OrderService.kt` | 로직 변경 | createOrder()에 재고 검증 로직 추가 |
| `InventoryClient.kt` | 신규 추가 | 새 클래스 |

### 영향받는 테스트
| # | 테스트 파일 | 테스트 메서드 | 영향 원인 | 깨질 가능성 | 수정 필요 |
|---|---|---|---|---|---|
| 1 | `OrderServiceTest.kt` | should_create_order_when_valid | 의존성 추가 (InventoryClient) | 높음 | Mock 추가 필요 |
| 2 | `OrderServiceTest.kt` | should_throw_when_invalid | 로직 변경 | 중간 | assertion 확인 필요 |
| 3 | `OrderControllerTest.kt` | should_return_201 | 간접 영향 | 낮음 | 변경 불필요 (Mock 격리) |

### 커버리지 갭
| # | 변경 항목 | 누락 케이스 | 우선순위 |
|---|---|---|---|
| 1 | createOrder() 재고 부족 분기 | 재고 부족 시 예외 발생 테스트 | 높음 |
| 2 | InventoryClient 전체 | Unit 테스트 없음 | 높음 |
| 3 | createOrder() 정상 경로 | 재고 검증 통과 후 주문 생성 테스트 | 보통 |

### 요약
- 변경된 프로덕션 파일: [N]개
- 영향받는 테스트 파일: [N]개 (깨질 가능성 높음 [N]개)
- 커버리지 갭: [N]개 (우선순위 높음 [N]개)

### 다음 단계
- 깨질 가능성이 높은 테스트를 먼저 실행하여 확인하세요.
- 커버리지 갭을 채우려면 `/test-write [대상 클래스]`를 실행하세요.
- 실패한 테스트가 있으면 `/test-debug [테스트 클래스]`로 원인을 분석하세요.
```

## 제약

- 프로덕션 코드와 테스트 코드를 수정하지 않는다. 분석 결과만 출력한다.
- 간접 영향 추적은 1단계(직접 의존하는 클래스)까지만 수행한다. 2단계 이상은 분석 비용 대비 정확도가 낮으므로 수행하지 않는다.
- 대상 파일을 찾을 수 없으면 "대상 파일을 찾을 수 없습니다. 파일 경로 또는 클래스명을 지정해 주세요."를 출력한다.
- develop/main/master 브랜치를 모두 찾을 수 없으면 "기준 브랜치를 찾을 수 없습니다. 대상 파일을 직접 지정해 주세요."를 출력한다.
- 변경 파일이 10개를 초과하면 파일별로 출력을 나누어 생성한다.
