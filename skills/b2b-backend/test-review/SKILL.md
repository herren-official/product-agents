기존 테스트 코드를 리뷰하고 개선점을 제시해라.

## 입력

$ARGUMENTS

- 인자가 파일 경로이면 해당 테스트 파일을 대상으로 한다.
- 인자가 클래스명이면 프로젝트에서 해당 테스트 클래스를 검색하여 대상으로 한다.
- 인자가 없으면 현재 브랜치와 develop 브랜치를 비교(`git diff develop...HEAD --name-only`)하여 추가/변경된 테스트 파일(`src/test/` 하위 .java, .kt)을 대상으로 한다. develop 브랜치가 없으면 main, master 순으로 시도한다.

## 절차

### Step 1: 테스트 파일과 프로덕션 코드 읽기
- 테스트 파일을 읽는다.
- 테스트 대상 프로덕션 코드를 찾아 함께 읽는다.
- 프로젝트의 테스트 컨벤션(다른 테스트 파일 1~2개)을 파악한다.

### Step 2: 커버리지 분석
- 현재 브랜치명을 `git branch --show-current`로 가져오고, `/`를 `-`로 치환한다.
- 프로젝트 루트의 `.claude/test/{branch-name}-test-plan.md` 파일이 존재하면 읽어 계획된 케이스 대비 구현 여부를 비교한다.

프로덕션 코드의 public 메서드/로직을 기준으로:
- 테스트된 케이스를 식별한다.
- 누락된 케이스를 도출한다 (정상, 경계값, 예외, 보안, 동시성).
- 테스트 계획 파일이 있으면 계획 대비 누락 케이스를 별도로 표시한다.

### Step 3: 품질 평가
다음 기준으로 각 테스트를 평가한다:

**구조:**
- Given-When-Then 구조가 명확한가?
- 한 테스트에 검증 포인트가 하나인가?
- 테스트명이 의도를 드러내는가?

**Mock 사용:**
- 불필요한 Mock이 있는가? (로직 없는 POJO를 Mock하는 등)
- Mock 검증이 구현 상세에 결합되어 있지 않은가?
- `any()` 남용으로 검증이 느슨하지 않은가?

**Fixture:**
- 테스트 데이터가 테스트 의도를 드러내는가?
- 매직 넘버/스트링 대신 의미 있는 변수명을 사용하는가?
- 테스트 간 fixture 공유로 상태 오염 위험이 있는가?

**안티패턴 탐지:**
- `@Transactional`로 롤백에 의존하여 실제 동작과 다른 결과를 내는가?
- `@SpringBootTest`가 Unit 테스트로 충분한 곳에 사용되었는가?
- `Thread.sleep()`으로 비동기 테스트를 처리하는가?
- try-catch로 예외를 검증하는가? (`assertThrows` / `shouldThrow` 사용 권장)
- `verify()`가 모든 Mock 호출을 빠짐없이 검증하여 깨지기 쉬운가?

### Step 4: 개선안 작성
문제마다 심각도(CRITICAL / WARNING / INFO)를 지정하고 구체적 수정 코드를 제시한다.

## 출력 형식

```markdown
## 테스트 리뷰 결과

### 대상
- 테스트 파일: `[경로]`
- 프로덕션 코드: `[경로]`
- 언어/프레임워크: [Java|Kotlin] / Spring Boot [버전]

### 커버리지 분석
| 메서드/로직 | 기존 테스트 수 | 누락 케이스 |
|---|---|---|
| methodA | 3 | 경계값(빈 리스트), 예외(DB 실패) |

### 품질 문제
| # | 심각도 | 위치 | 문제 | 개선안 |
|---|--------|------|------|--------|
| 1 | CRITICAL | L25-30 | @SpringBootTest를 단위 테스트에 사용 | @ExtendWith(MockitoExtension.class)로 변경 |
| 2 | WARNING | L42 | Thread.sleep(1000)으로 비동기 대기 | Awaitility 사용 권장 |
| 3 | INFO | L15 | 테스트명이 의도를 드러내지 않음 | should_xxx_when_yyy 패턴 적용 |

### 수정 코드 (CRITICAL만)
\`\`\`[java|kotlin]
// 수정 전
...
// 수정 후
...
\`\`\`

### 요약
- 전체 테스트 수: [N]개
- CRITICAL: [N]개 / WARNING: [N]개 / INFO: [N]개
- 누락 케이스: [N]개
```

## 제약

- 프로덕션 코드를 수정하지 않는다.
- 기존 테스트의 스타일을 존중하되, 안티패턴은 명확히 지적한다.
- 대상 파일을 찾을 수 없으면 "대상 테스트 파일을 찾을 수 없습니다. 파일 경로 또는 클래스명을 지정해 주세요."를 출력한다.
- develop/main/master 브랜치를 모두 찾을 수 없으면 "기준 브랜치를 찾을 수 없습니다. 대상 파일을 직접 지정해 주세요."를 출력한다.

## 다음 단계 안내

출력 끝에 다음을 추가한다:
```
### 다음 단계
- 수정 코드를 적용하려면 CRITICAL 항목부터 반영하세요.
- 누락 케이스를 추가하려면 `/test-write [대상 클래스]`를 실행하세요.
- 실행 검증이 필요하면 "테스트 검증해줘"라고 요청하세요.
```
