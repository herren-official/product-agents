대상 코드를 분석하여 작성해야 할 테스트 케이스 목록을 도출해라. 코드는 작성하지 않는다. 도출 결과를 테스트 계획 파일로 저장한다.

## 입력

$ARGUMENTS

- 인자가 파일 경로이면 해당 파일을 대상으로 한다.
- 인자가 클래스명이면 프로젝트에서 해당 클래스를 검색하여 대상으로 한다.
- 인자가 없으면 현재 브랜치와 develop 브랜치를 비교(`git diff develop...HEAD --name-only`)하여 추가/변경된 프로덕션 코드 파일(.java, .kt)을 대상으로 한다. 테스트 파일(`src/test/`)은 대상에서 제외한다. develop 브랜치가 없으면 main, master 순으로 시도한다.

## 절차

### Step 1: 대상 코드 파악
- 대상 파일을 읽고 public 메서드, 비즈니스 로직, 의존성을 파악한다.
- `src/test/` 하위에서 대상 클래스명 + `Test` 접미사로 기존 테스트 파일을 검색한다. 있으면 함께 읽어 이미 작성된 케이스를 파악한다.

### Step 2: 테스트 유형 분류
각 메서드/로직에 대해 다음 유형으로 분류한다:
- **Unit**: 외부 의존성 없이 로직만 검증
- **Slice**: `@WebMvcTest`, `@DataJpaTest` 등 특정 레이어 검증
- **Integration**: `@SpringBootTest` 또는 Testcontainers로 전체 흐름 검증

### Step 3: 테스트 케이스 도출
각 메서드/로직에 대해 다음 관점으로 케이스를 도출한다:
- 정상 경로 (happy path)
- 경계값 (빈 리스트, null, 최대값)
- 예외/에러 (비즈니스 예외, 시스템 예외)
- 보안 (권한 없는 접근, 인증 미수행)
- 동시성 (대상 코드에 `@Version`, `synchronized`, `Lock`, `AtomicXxx`, `CompletableFuture` 등이 있는 경우: Optimistic Lock, Race condition)

## 출력 형식

```markdown
## 테스트 케이스 목록

### 대상: `[클래스명]`
- 기존 테스트 파일: `[경로]` 또는 없음
- 기존 커버리지: [N]개 케이스 존재

### Unit 테스트
| # | 메서드 | 케이스 | 검증 포인트 | 기존 여부 |
|---|--------|--------|-------------|-----------|
| 1 | methodA | 정상 입력 시 결과 반환 | 반환값 일치 | 없음 |
| 2 | methodA | null 입력 시 예외 발생 | IllegalArgumentException | 있음 |

### Slice 테스트
| # | 대상 | 케이스 | 어노테이션 | 기존 여부 |
|---|------|--------|-----------|-----------|

### Integration 테스트
| # | 시나리오 | 케이스 | 필요 인프라 | 기존 여부 |
|---|----------|--------|-----------|-----------|

### 요약
- 총 도출: [N]개 (Unit [N] / Slice [N] / Integration [N])
- 기존 존재: [N]개
- 신규 필요: [N]개
```

## Step 4: 테스트 계획 파일 저장

도출된 케이스 목록을 프로젝트 루트의 `.claude/test/{branch-name}-test-plan.md` 파일로 저장한다.
- 현재 브랜치명을 `git branch --show-current`로 가져온다.
- 브랜치명에 `/`가 포함되면 `-`로 치환한다. (예: `feature/order` → `feature-order`)
- `.claude/test/` 디렉토리가 없으면 생성한다.
- 동일 브랜치의 기존 파일이 있으면 덮어쓴다.
- 파일 상단에 메타 정보를 HTML 주석으로 추가한다:

```markdown
<!-- generated-by: /test -->
<!-- generated-at: [YYYY-MM-DD HH:mm] -->
<!-- branch: [브랜치명] -->
<!-- target: [대상 파일 경로 목록] -->
```

저장 완료 후 파일 경로를 사용자에게 안내한다.

## 제약

- 코드를 생성하지 않는다. 목록만 출력하고 테스트 계획 파일에 저장한다.
- 프레임워크 버전(Spring Boot 2.x/3.x)에 따른 어노테이션 차이가 있으면 비고에 명시한다.
- 대상 파일을 찾을 수 없으면 "대상 파일을 찾을 수 없습니다. 파일 경로 또는 클래스명을 지정해 주세요."를 출력한다.
- develop/main/master 브랜치를 모두 찾을 수 없으면 "기준 브랜치를 찾을 수 없습니다. 대상 파일을 직접 지정해 주세요."를 출력한다.
- 대상 파일이 5개를 초과하면 파일별로 출력을 나누어 생성한다.

## 다음 단계 안내

출력 끝에 다음을 추가한다:
```
### 다음 단계
- 케이스 도출 → 작성 → 실행 → 디버깅 → 리뷰를 자동으로 수행하려면 `/test-all [대상 클래스]`를 실행하세요.
- 신규 테스트를 작성만 하려면 `/test-write [대상 클래스]`를 실행하세요.
- 기존 테스트를 리뷰하려면 `/test-review [테스트 클래스]`를 실행하세요.
```
