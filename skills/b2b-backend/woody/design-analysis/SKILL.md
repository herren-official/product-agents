# Design Analysis (기획-디자인 대조 분석)

## Trigger Keywords

이 스킬은 다음 키워드로 활성화됩니다:
- "기획 분석", "기획서 분석", "작업 리스트업"

## Step 1: 입력 수집

사용자에게 아래 정보를 AskUserQuestion으로 수집한다. 이미 제공된 항목은 건너뛴다.

```
필수 입력:
1. 노션 기획서 URL
2. 저장 경로 (예: ai/claude/woody/2026-task/epic-xxx)

선택 입력:
3. Figma 디자인 URL (없으면 노션만 분석)
4. 담당 범위 (전체 백엔드 / 특정 영역)
5. 에픽/티켓 번호 (예: GBIZ-XXXXX)
```

**입력 수집 질문 예시:**
```
기획-디자인 대조 분석을 시작합니다.

1. 노션 기획서 URL을 알려주세요.
2. Figma 디자인 URL이 있나요? (없으면 노션만 분석합니다)
3. 결과를 저장할 경로를 지정해주세요. (예: ai/claude/woody/2026-task/epic-xxx)
4. 담당 범위는 어디까지인가요? (전체 백엔드 / 특정 영역)
```

## Step 2: 노션 기획서 분석

### 2-1. 노션 페이지 가져오기

```
notion-fetch MCP 도구로 페이지 내용을 가져온다.
결과가 큰 경우(50K+ 토큰) 파일로 저장 후 chunk 단위로 읽는다.
이미지 URL은 무시하고 텍스트 콘텐츠만 추출한다.
```

### 2-2. 추출 항목

| 항목 | 설명 |
|------|------|
| 기능 요구사항 | 체크박스 항목 포함, 섹션별 분류 |
| 비기능 요구사항 | 성능, 보안, 컴플라이언스 |
| 외부 연동 | 벤더, API 문서 링크, 단가 정보 |
| 사용자 시나리오 | 주요 유저 플로우 |
| 제약 사항 | 법적 제약, 기술 제약 |
| 메뉴 구조 | 화면 계층 구조 |

### 2-3. 백엔드 작업 리스트 생성

기능 요구사항을 아래 구조로 정리한다:

```markdown
### Phase N: {영역명}

#### {번호}. {기능명} (기획서 섹션 번호)
- [ ] {백엔드 작업 항목}
- [ ] {백엔드 작업 항목}
```

**분류 기준:**
- API 엔드포인트가 필요한 항목
- 비즈니스 로직/검증이 필요한 항목
- 배치/스케줄러가 필요한 항목
- 외부 연동이 필요한 항목
- DB 설계가 필요한 항목

### 2-4. README.md 저장

아래 구조로 `{저장경로}/README.md`에 저장:

```markdown
# {기능명}

## 기획 문서
- [노션 기획서]({URL})

## 개요
{1~2줄 요약}

## 메뉴 구조
{있으면 포함}

---

## 백엔드 작업 리스트
{Phase별 체크리스트}

---

## 비기능 요구사항
{테이블}

## 단가/비용 정보
{있으면 포함}

## 벤더 연동 현황
{있으면 포함}

## 개발 문서
{API 문서 링크 등}
```

## Step 3: Figma 디자인 분석 (URL 제공 시)

### 3-1. URL 파싱

```
URL: figma.com/design/:fileKey/:fileName?node-id=:int1-:int2
→ fileKey: :fileKey
→ nodeId: :int1::int2 ("-" → ":" 변환)

branch URL: figma.com/design/:fileKey/branch/:branchKey/:fileName
→ fileKey: :branchKey (branch key를 사용)
```

### 3-2. 메타데이터 수집

```
1. get_metadata(fileKey, nodeId)로 페이지 전체 구조를 가져온다
2. 최상위 섹션(depth 1) 목록 추출: id, name
3. 하위 섹션(depth 2) 목록 추출: id, name
```

**추출 스크립트:**
```python
# 최상위 섹션 (canvas 직접 자식)
pattern: ^  <section id="..." name="..."
# 하위 섹션 (최상위 섹션의 자식)
pattern: ^    <section id="..." name="..."
```

### 3-3. 스크린샷 수집 (선택)

```
get_screenshot(fileKey, nodeId)로 주요 화면별 스크린샷을 가져온다.
- 무료 플랜 한도 주의: 최상위 섹션 3~5개 우선
- 한도 초과 시 메타데이터 기반으로 분석 진행
- 에러 발생 시 graceful하게 계속 진행
```

## Step 4: 대조 분석 (GAP Analysis)

### 4-1. 매핑 규칙

```
1. 노션 기능 항목의 이름/키워드 ↔ Figma 섹션 이름을 대조
2. 1:1 매핑 안 되면 상위/하위 섹션까지 확인
3. 백엔드 전용 작업(배치, API 연동)은 "UI 불필요"로 분류
4. Figma에만 있는 화면은 "기획 확인 필요"로 분류
```

### 4-2. 분석 카테고리

| 상태 | 의미 | 액션 |
|------|------|------|
| 🔴 Figma에만 존재 | 디자인은 있으나 기획서에 상세 없음 | 기획팀 확인 필요 |
| 🟡 노션에만 존재 | 기획서에는 있으나 Figma에 화면 없음 | 디자인팀 확인 또는 백엔드 전용 |
| 🟢 양쪽 대응 | 기획과 디자인이 잘 매핑됨 | 정상 |

### 4-3. NOTION-FIGMA-GAP-ANALYSIS.md 저장

```markdown
# 노션 기획서 vs Figma 디자인 대조 분석

## 분석 기준
- **노션**: {기획서 제목} - 기능 요구사항
- **Figma**: {파일명} > {페이지명} 섹션 구조

---

## 1. 매핑 테이블: 노션 기능 ↔ Figma 화면

### {영역명}
| 노션 기능 요구사항 | Figma 섹션 (Node ID) | 상태 |
|---|---|---|
| {기능명} | {섹션명} ({nodeId}) | ✅/⚠️ |

---

## 2. GAP 분석

### 🔴 Figma에만 있는 항목
| 항목 | 설명 | 영향도 |
|------|------|--------|

### 🟡 노션에만 있는 항목
| 항목 | 노션 위치 | 설명 |
|------|----------|------|

### 🟢 양쪽 잘 대응됨
| 영역 | 평가 |
|------|------|

---

## 3. 확인 필요 사항
### 기획팀/디자인팀 확인 필요
1. ...

### 백엔드 추가 검토
1. ...

---

## 4. 요약
| 지표 | 결과 |
|------|------|
| 노션 기능 항목 수 | N개 |
| Figma 섹션 수 | M개 |
| 완전 매핑 | X/N (Y%) |
| Figma 추가 화면 | A개 |
| 노션만 있는 항목 | B개 |
| 전체 정합성 | 높음/보통/낮음 |
```

## Step 5: 결과 보고

터미널에 아래 형식으로 요약을 출력한다:

```
## 분석 완료

| 지표 | 결과 |
|------|------|
| 노션 기능 항목 | N개 영역 |
| Figma 섹션 | M개 |
| 매핑률 | Y% |

### 저장된 파일
- {저장경로}/README.md - 백엔드 작업 리스트
- {저장경로}/NOTION-FIGMA-GAP-ANALYSIS.md - 대조 분석 보고서

### 기획/디자인팀 확인 필요 (N건)
1. ...
2. ...
```

## Figma 없이 실행한 경우

Step 3, 4를 건너뛰고 Step 2 결과만 README.md로 저장한다.
이후 Figma URL을 받으면 Step 3, 4만 추가 실행할 수 있다.

```
사용자: "여기 Figma도 있는데 대조해줘"
→ 기존 README.md를 읽고 Figma만 분석하여 GAP 보고서 추가 생성
```

## 참고 사항

### MCP 도구 의존성
| 도구 | 필수 | 용도 |
|------|------|------|
| `notion-fetch` | ✅ | 노션 페이지 내용 조회 |
| `figma/get_metadata` | 선택 | Figma 섹션 구조 파악 |
| `figma/get_screenshot` | 선택 | 화면 스크린샷 수집 |

### 대용량 노션 문서 처리
```
1. notion-fetch 결과가 파일로 저장된 경우
2. python3으로 JSON 파싱하여 text 필드 추출
3. 이미지 URL(S3) 필터링
4. chunk 단위(5000~7000자)로 읽기
```

### Figma 호출 한도 대응
```
1. get_metadata → 구조 파악 (1회)
2. get_screenshot → 주요 화면 3~5개 (3~5회)
3. 한도 초과 시 → 메타데이터 기반으로 분석 계속
4. 개별 화면은 Figma URL + node-id 변경하여 직접 확인 안내
```
