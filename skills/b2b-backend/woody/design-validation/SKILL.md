# 설계 검증 (Design Validation)

## 역할

아키 산출물(`10_architect_design.md`)의 품질을 자동 검증한다.
1. DB 네이밍 규칙 위반 검출
2. 설계 정합성 검증 (흐름, DB, 상태, 에러 처리)

## 워크플로우

### Step 1: DB 네이밍 자동 검증

산출물 파일에서 Grep으로 위반 패턴 검출:

| 위반 패턴 | 올바른 형식 | 검색 정규식 |
|----------|-----------|-----------|
| `shop_no` | `shopno` | `shop_no` |
| `customer_no` | `custno` | `customer_no` |
| `created_at` | `regdate` | `created_at` |
| `updated_at` | `moddate` | `updated_at` |
| `*_at` (단어 끝) | `*_datetime` | `_at\b` (DDL/엔티티 컨텍스트에서) |

**처리:**
- 위반 0건 → Step 2로 진행
- 위반 발견 → 위반 목록 + 수정 제안 보고. 아키에게 수정 요청 후 재검증.

### Step 2: 설계 정합성 검증

산출물 전체를 읽고 아래 관점에서 검증:

**2.1 아키텍처 흐름 검증:**
- 전체 플로우에 빈 구간이 없는지 (데이터가 끊기는 구간)
- 모듈 간 통신 방식이 명확한지 (SQS, REST API)
- 비동기 흐름에서 실패 시 보상 로직이 있는지

**2.2 DB 참조 관계 검증:**
- 신규 테이블 간 FK/참조 관계 누락
- 인덱스가 조회 패턴에 맞는지
- 테이블 접두사 통일 여부

**2.3 상태 전이 검증:**
- 상태 머신에 빠진 전이가 없는지
- 터미널 상태에서 복귀 경로가 정의되어 있는지 (재발송 등)
- 동시 상태 변경 시 경합 방지 방법이 있는지

**2.4 에러/실패 처리 검증:**
- SQS 메시지 처리 실패 시 DLQ 전략
- 벤더 API 장애 시 대응
- 타임아웃 처리 (SENDING 상태 정체 등)
- 차감 원복 누락 가능성

### Step 3: 결과 보고

심각도별 분류:

| 심각도 | 기준 | 처리 |
|--------|------|------|
| **Critical** | 실행 차단, 데이터 정합성 위험 | 아키에게 수정 요청, 재검증 필수 |
| **Major** | 상당한 재작업 필요 | 아키에게 수정 요청 권장 |
| **Minor** | 기능적 문제 없으나 개선 필요 | Woody님 판단 |

**보고 형식:**
```
[설계 검증 결과]

## DB 네이밍 검증
- 위반: N건 / 통과: M건

## 정합성 검증
### Critical (N건)
1. {이슈} — {근거} — {수정 제안}

### Major (N건)
1. {이슈} — {근거} — {수정 제안}

### Minor (N건)
1. {이슈}

## 판정: PASS / PASS-WITH-RESERVATIONS / FAIL
```
