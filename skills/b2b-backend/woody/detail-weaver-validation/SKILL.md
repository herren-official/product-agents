# 디테일러 산출물 교차 검증 (Detail Weaver Validation)

## 역할

디테일러가 작성한 TASK 구현 명세서를 **메타 정합성 관점**에서 검증한다. 코드체커(팩트 대조)의 사각지대인 **TASK 간 책임 분담·범위 경계·심볼 추적·설계서 섹션 매핑**을 담당한다.

## 워크플로우

### Step 1: 입력 수집

1. 검증 대상 TASK 문서 Read (예: `task/TASK-XX.md`)
2. 설계서 `10_architect_design.md` — 대상 TASK가 반영해야 할 섹션 식별
3. TASK 분해 문서 `11_tasker_tasks.md` — 선행/후속/형제 TASK 관계 파악
4. 리뷰 결정 `12_woody_review.md` — 확정 정책(P-xx) 목록 확보
5. 선행 TASK 산출물 — 확정된 심볼 목록 수집
6. 후속/형제 TASK 골격 — 중복/충돌 감지용
7. (선택) 코드체커 결과 — 있으면 상보 참조

### Step 2: 8가지 검증 항목 (v2 — 실전 축적 반영)

#### 2.1 선행 TASK 심볼 참조 정확성 + **grep 실측 필수 (v2 강화)**

- 선행 TASK에서 확정된 Enum/sealed class/DTO/인터페이스 목록
- 본 TASK가 인용한 심볼 이름·필드·위치 대조
- **누락**(선행 심볼 있으나 본 TASK 미인용) / **오타**(이름 불일치) / **재정의**(본 TASK가 선행과 충돌하는 동명 심볼 신설) 감지
- **[v2 강화]** 본 TASK의 COMMENT·docstring·Enum 인용 문자열에 등장한 심볼 이름을 **선행 TASK 파일 전체 grep**으로 실제 정의 존재 확인 필수
  - 실전 사례: TASK-02가 `DeductionLogType`을 DDL COMMENT에 선행 Enum처럼 인용했으나 TASK-01에 정의 부재 — Critical-1 발견 근거
  - grep 패턴: `grep -n "^enum class {심볼명}\|sealed class {심볼명}\|interface {심볼명}\|data class {심볼명}" {선행 TASK 경로}`

#### 2.2 후속/형제 TASK 중복·충돌 + **카운트 정합성 (v2 신설)**

- 본 TASK에서 정의한 파일/심볼이 **후속 TASK 골격에도 중복** 명시되어 있는가? → 중복 할당
- 본 TASK에서 미정의했지만 **후속 TASK 골격에 이미 할당**되어 있는 항목 → 본 TASK 범위 외인데 디테일러가 누락으로 착각할 여지
- 형제 TASK(병렬 실행)와 파일 경로 충돌
- **[v2 신설]** **카운트·숫자 정합성 교차 대조**: 본 TASK의 "N개 테이블/파일/엔티티" 표기 vs `11_tasker_tasks.md` + 형제/후속 TASK 숫자 일치 여부
  - 실전 사례: TASK-02 "DDL 7종" vs `11_tasker_tasks.md`·TASK-06 "6종" 드리프트 (P-46 deduction_log 추가 후 역전파 누락) — Major 발견
  - **본 TASK 내부 숫자 무결성**: 헤더 "N개"와 실제 열거 개수 일치 여부도 체크 (예: "15개" 표기인데 실제 17개 나열)

#### 2.3 범위 경계(out-of-scope) 명시 + **고아 Port 탐지 (v2 신설)**

- 본 TASK 문서에 **"이 TASK가 하지 않는 것"**, **"범위 외"**, **"TASK-YY로 위임"** 같은 명시적 섹션 존재 여부
- 없으면 Major (후속 검증에서 혼동·오탐의 주원인)
- **[v2 신설]** **3열 표(항목/이유/위임 대상) 스타일 권장** — 코드체커 오탐 구조적 차단 효과 입증 (TASK-03 사례)
- **[v2 신설]** **고아 Port 탐지**: Port 인터페이스를 본 TASK에서 선언했는데 **같은 TASK 내 Adapter/소비처 부재**인 경우, "왜 Port만 두는지" 의도가 docstring에 명시됐는지 확인
  - 실전 사례: TASK-03의 `SlackAlertSender` Port 선언했으나 `SenderSlackAlertService`가 `SlackClient` 직접 주입 — placeholder 근거 명시 전에는 Minor

#### 2.4 설계서 섹션 매핑 + **번호 유일성/드리프트 (v2 신설)**

- 설계서에서 대상 TASK와 관련된 섹션 식별
- 각 섹션의 핵심 항목이 본 TASK에 반영됐는지 체크
- **누락**: 설계서에 있는데 본 TASK에 빠짐 → Critical 가능성
- **과잉**: 본 TASK에 있는데 설계서에 근거 없음 → Major
- **[v2 신설]** **설계서 섹션 번호 유일성 검증**: `grep "^### N\." 설계서` 결과의 번호가 중복되는지 체크. 중복이면 디테일러 인용도 오염될 확률 높음
  - 실전 사례: 설계서 `4.7 notification_template` + `4.7 notification_campaign_performance` 중복 → TASK-02 체크리스트가 구 번호로 오염 → Major 발견
- **[v2 신설]** **명칭 드리프트 자동 감지**: 설계서의 `*.kt` 파일명 vs 본 TASK의 `*.kt` 파일명을 전수 대조 → 불일치 시 디테일러가 리네이밍 주석을 남겼는지 확인
  - 실전 사례: 설계서 `SendHistoryService.kt` → TASK-03 `SendHistoryCasService.kt` 리네임 (의미 개선). 주석 없으면 Minor → 주석 있으면 OK
- **[v2 신설]** **상위 문서 → 하위 TASK 참조 전파 검증**: 설계서 섹션 번호·정책 번호가 최근 수정됐다면 본 TASK 내 참조도 최신 번호인지 grep
  - 실전 사례: 설계서 4.7→4.8→4.9 재번호 후 TASK-02 본문 "4.2~4.7" 잔존 → LOOP5 발견

#### 2.5 P-xx 정책 번호 인용 정확성

- 본 TASK가 인용한 P-xx 번호가 **설계서 10장 / 12_woody_review의 정책 내용과 일치**하는가
- 번호 혼동, 버전 차이(P-09 재정의 이전/이후) 감지
- **[v2 강화]** 재정의된 정책(예: P-09 "물리 Enum 별개" 개정)의 **구 해석 인용 여부** 감지

#### 2.6 인터페이스 시그니처 일관성

- Producer TASK에서 정의한 DTO/인터페이스 필드 ↔ Consumer TASK에서 사용하는 형태 대조
- SQS 메시지 필드, REST API 요청/응답, Port 인터페이스 시그니처
- **[v2 강화]** snake_case ↔ camelCase 미스매치 같은 표기 불일치도 체크

#### 2.7 완료 조건 검증 가능성 + **조건부 TODO 플래그 (v2 신설)**

- 각 완료 조건이 **구체적 검증 방법**(테스트 클래스명, gradle 커맨드, 측정 가능 기준)을 포함하는지
- 추상적/주관적 완료 조건이 남아있는지
- **[v2 신설]** **조건부 TODO 탐지**: "PR 직전 확정", "TASK-XX 실측 후 결정", "운영팀 확인 후" 같은 후행 의존 TODO를 자동 플래그 → Minor 수집 (반복 발생 시 Major 승격)
  - 실전 사례: TASK-06 "TASK-01 실측 후 PR 직전 확정" 조건부 TODO가 루프4에서 Minor로 재지적

#### 2.8 논리/물리 소유 구분 (v2 신설)

- 본 TASK가 "논리 소유자(담당 모듈)" vs "물리 실행자(실제 파일 생성 위치)" 구분을 명시하는가?
  - 실전 사례: TASK-02는 "DDL 논리 소유 = orchestrator/batch" vs "물리 Liquibase 실행 = b2b-backend master.xml" 명시 → 루프 검증에서 안전
- 구분 없으면 하위 TASK(엔티티 매핑 등)가 어느 모듈에 배치해야 할지 혼동

#### 2.9 외부 의존 결정 완결성 (v3 신설 — 2026-04-26)

> 본 TASK가 외부 API에 의존하는 경우, 외부 결정이 완료된 상태인지 검증.
> Decision Gate 표준(`.claude/docs/decision-gate.md`)을 위버 단계에 통합.

**검증 항목**:
- 본 TASK가 외부 API/벤더에 의존하는가? (인포뱅크 / LG CNS / BizTalk / S3 / Redis 등)
- 의존 시 다음 결정이 명시·해결됐는가?
  - **호출 흐름**: 사전 업로드 vs URL 전달 vs base64 인라인
  - **라이프사이클**: TTL / 갱신 정책 / 캐시
  - **동시성**: 분산 락 / race condition 처리
  - **운영 전제**: 계정 / API key / 테스트 환경 / 영업 컨택
- 결정 완료 = `DECISION-XX.md` 양식 존재
- 결정 미완 = 차단 R-XX로 등록되어야 함

**판정 강화**:
- 외부 의존 결정 미완 1건 = **Major** (Critical 가능)
- 본 TASK가 차단 R-XX 영향 범위에 있는데 **DEFERRED 상태로 처리한 명시적 합의 없음** = **Critical**
- 운영 전제 매트릭스(설계서 9.X) 미참조 = Major

**실전 사례 회고 (2026-04-26)**:
- TASK-04(인포뱅크) 디테일러 진입 시도 시 R-08(이미지 업로드 옵션) 미해결 → 위버가 차단했어야 함
- TASK-05(LG CNS) 디테일러 진입 시도 시 R-09(톡드림 MMS 영업 미팅) 미해결 → 위버가 차단했어야 함
- 본 항목 신설로 향후 동일 누락 차단

### Step 3: 심각도 분류 (v2)

| 심각도 | 기준 | 처리 |
|--------|------|------|
| **Critical** | TASK 간 중복 할당, **선행 심볼 정의 부재(grep 0건)**, 설계서 섹션 통째 누락 | 디테일러 재실행 필수 |
| **Major** | 범위 경계 미명시, 인터페이스 불일치, 정책 번호 혼동, **카운트 드리프트**, **설계서 번호 중복**, **상위 문서 참조 전파 누락** | 디테일러 재실행 권장 (메인 패치로도 해결 가능) |
| **Minor** | 문구 개선, 표기 통일, 완료 조건 구체화, **명칭 드리프트 주석 누락**, **조건부 TODO 미해결(첫 회)**, **고아 Port placeholder 근거 부족** | 기록만 (메인 패치) |

**[v2 규칙] 반복 지적 승격**: 같은 항목이 루프 2회 이상 지적되면 심각도 1단계 승격 (Minor → Major → Critical). 디테일러가 재실행에도 고치지 못하는 구조적 이슈 구분.

## 학습 포인트 (v2 실전 축적)

### 디테일러 공통 약점 8가지 (검증 체크리스트)

1. **선행 TASK 심볼 grep 검증 부재**: COMMENT 문자열의 Enum 이름이 선행 TASK에 실제 정의 존재 여부 미확인
2. **카운트 변경 역전파 미흡**: 테이블/파일 개수 변경이 형제/후속 TASK에 전파 누락
3. **설계서 섹션 번호 중복 무검증**: 설계서를 신뢰해 중복 번호 그대로 인용
4. **조건부 TODO 미해결**: "PR 직전 확정"이 반복 지적됨
5. **상위 문서 → 하위 TASK 전파 누락**: 설계서 재번호 후 TASK 본문 참조 스테일
6. **명칭 드리프트**: 의미 개선 리네이밍 시 주석 누락
7. **고아 Port**: Port 선언했으나 소비처 없음 (placeholder 근거 부족)
8. **문서 내 숫자 불일치**: 헤더 "N개" vs 실제 열거 개수

### 디테일러 강점 (유지·확산)

1. **범위 외 3열 표 (항목/이유/위임 대상)**: 코드체커 오탐 구조적 차단 효과 입증 → 디테일러 필수 섹션으로 승격 권장
2. **DDL ↔ 엔티티 1:1 매핑표**: Producer/Consumer 자기 문서화
