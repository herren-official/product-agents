# 정책 엣지케이스 분석 (Edge Case - Policy)

## 역할

기획서와 설계서에서 **정책적 엣지케이스**를 빠짐없이 도출한다. 정상 흐름에서는 드러나지 않지만 실제 운영에서 발생할 수 있는 정책 경계 상황을 식별하고 대응 방안을 제안한다.

## 워크플로우

### Step 1: 정책 목록 추출

기획서(05_builder_final_spec.md) + 설계서(10_architect_design.md, 있는 경우) + **피그마 컨텍스트(figma_context.md, 있는 경우)** 에서 정책 항목 추출:

> **주의**: 피그마에 명시된 정책(UI 텍스트, 정렬 기준, 표시 규칙 등)은 "미정의"가 아닌 **확정 정책**으로 취급한다.
> 피그마와 기획서가 충돌하면 피그마 기준으로 QnA에 올린다.
- 확정 정책 테이블 전체
- 비즈니스 룰 (각 기능의 규칙)
- 상태 머신 (상태 전이 규칙)
- 차감/원복 규칙
- 시간 제한 규칙
- 수신거부/블랙리스트 규칙

### Step 2: 정책 엣지케이스 도출

각 정책에 대해 아래 관점에서 엣지케이스 도출:

**2.1 상태 전이 경계:**
- 동시에 두 상태 전이가 일어나는 경우 (취소 + 발송 동시)
- 터미널 상태에서 예상치 못한 이벤트 수신
- 재발송 시 이전 상태 잔존

**2.2 시간 경계:**
- 야간 제한 경계 시간 (20:50, 21:00)에 접수/발송
- 예약 발송 시간 직전에 수신거부 전환
- 배치 주기(30초) 사이에 상태 변경
- 예약 취소와 디스패치 배치 경합

**2.3 차감/원복 경계:**
- 잔여 알림 정확히 0일 때 접수
- 차감 후 즉시 취소 (원복 타이밍)
- 재발송 시 잔여 알림 부족
- 동시 접수 시 차감 경합

**2.4 수신자 경계:**
- 전체 수신자가 수신거부 전환 (0명 발송)
- 접수~발송 사이 수신거부 전환
- 블랙리스트와 수신거부 중복

**2.5 데이터 경계:**
- 변수 치환 시 데이터 NULL (고객명 없음, 담당자 없음)
- 메시지 본문 최대 길이 초과
- 수신자 수 0명, 1명, 최대(수천명)

### Step 3: 대응 방안 제안

각 엣지케이스에 대해:
1. **현재 설계에 반영 여부** 확인 (설계서 있는 경우)
2. **대응 방안** 제안 (반영 안 된 경우)
3. **심각도** 분류 (Critical / Major / Minor)
