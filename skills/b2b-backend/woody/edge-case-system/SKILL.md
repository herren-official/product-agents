# 시스템 엣지케이스 분석 (Edge Case - System)

## 역할

아키텍처 설계서에서 **시스템 엣지케이스**를 빠짐없이 도출한다. 런타임 장애, 동시성, 인프라 제약 등 시스템 관점에서 발생할 수 있는 비정상 상황을 식별하고 대응 방안을 제안한다.

## 워크플로우

### Step 1: 아키텍처 구성 요소 추출

설계서(10_architect_design.md)에서 시스템 구성 요소 목록 추출:
- 모듈 (Backend, Orchestrator, Sender, Batch)
- 통신 (SQS, REST API)
- DB (트랜잭션, 락, 인덱스)
- 외부 벤더 (인포뱅크, LG CNS, BizTalk)
- 배치 스케줄러 (주기, 동시 실행)
- 상태 머신 (상태 전이 + 낙관적 락)

### Step 2: 시스템 엣지케이스 도출

각 구성 요소에 대해 아래 관점에서 엣지케이스 도출:

**2.1 장애/복구:**
- 서버 재시작 (발송 중간에 재시작 → 미완료 건 처리)
- DB 커넥션 끊김 (차감 후 커밋 전 장애)
- 벤더 API 장시간 장애 (대기 건 무한 적체)
- OOM / 메모리 부족

**2.2 동시성/경합:**
- 같은 Campaign 동시 취소 + 디스패치
- 같은 SendHistory 동시 상태 업데이트 (낙관적 락 충돌)
- 같은 Shop 동시 차감 (원자적 UPDATE 경합)
- 리포트 Polling과 타임아웃 배치 동시 실행

**2.3 SQS:**
- 메시지 유실 (at-least-once 보장이지만 극단적 경우)
- 중복 수신 (동일 sendHistoryId 2회 처리)
- 순서 역전 (ASSEMBLED 전에 SENDING 도착)
- DLQ 적체 (처리 불가 메시지 누적)
- visibility timeout 초과 (처리 중 재배달)

**2.4 배치:**
- 동일 배치 2회 실행 (SKIP LOCKED 없으면 중복)
- 배치 실행 시간이 주기보다 긴 경우 (30초 주기인데 처리 45초)
- 배치 장애로 미실행 (리포트 미수신 → SENDING 정체)

**2.5 외부 벤더:**
- 인포뱅크 Rate Limit 초과 (200 TPS)
- 인포뱅크 리포트 미반환 (3일 후 자동 삭제)
- LG CNS dev에서도 실제 발송 (테스트 주의)
- 벤더 응답 지연 (타임아웃 vs 재시도)
- 벤더 응답 형식 변경 (API 버전 업)

**2.6 데이터/용량:**
- 대량 발송 (5,000명) 시 SendHistory 벌크 INSERT 성능
- 리포트 1,000건 벌크 처리 시 DB 커넥션 풀 고갈
- attachment JSON 크기 제한 (캐러셀 등 복잡한 구조)
- 장기 운영 시 send_history 테이블 비대화

**2.7 외부 API 결정 흐름 (v2 신설 — 2026-04-26)**

> 외부 API 사용 시 "어떻게 호출할지"가 결정됐는지 검증.
> 미결정이면 후속 단계 진입 차단 (Decision Gate 적용).

- **호출 흐름 결정 여부**:
  - 사전 업로드 vs URL 전달 vs base64 인라인 — 어느 방식?
  - 인포뱅크 카카오 브랜드 = imgUrl 사전 업로드 필수 (외부 URL 불가)
  - 만약 미결정 = R-XX 등록 + 차단
- **벤더 식별자 정확성**:
  - "LG CNS" 표기만으로는 불충분 (톡드림? 비즈뿌리오? 다른 자회사?)
  - 실제 endpoint(예: `talkapi.lgcns.com`) 코드베이스 grep으로 교차 확인
  - 비즈뿌리오(인포뱅크 자회사) ≠ 톡드림(LG CNS 자체) 같은 혼동 사례 방지
- **공개 API 부재 가능성**:
  - 모든 외부 API가 공개 문서 있다고 가정 금지
  - 폐쇄형 B2B 서비스(예: LG CNS 톡드림)는 영업 미팅 필수
  - 미확보 시 R-XX 등록

**2.8 외부 의존 라이프사이클 (v2 신설)**

- **TTL/만료 정책**: fileKey/imgUrl/토큰 만료 시간이 결정됐는가?
  - 인포뱅크 `expired` 필드 동적 응답
  - 비즈뿌리오 24h 고정 만료 (참고)
- **갱신 전략**: 만료 임박 시 재발급 자동/수동?
  - Lazy refresh (사용 시점)
  - Pre-warm 배치 (만료 1h 전 일괄 갱신)
- **재발급 멱등성**: 동일 자원 재업로드 시 동일 키 반환? 새 키?
- **캐시 정책**: vendor_file_cache 같은 매핑 테이블 필요한가?

**2.9 외부 의존 동시성 (v2 신설)**

- **동일 자원 동시 호출 race**: 같은 이미지에 4명이 동시 발송 시?
  - 분산 락 미적용 → N번 업로드
  - 해결: Redis SETNX 또는 DB SELECT FOR UPDATE
- **이중 발급 방지**: 락 미획득 시 짧은 대기 후 캐시 재조회 (singleflight 패턴)
- **벤더 Rate Limit 동시성**: 200 TPS 제약을 다중 인스턴스가 어떻게 분배하는가?

**2.10 운영 전제 조건 (v2 신설)**

> "기술적으로 호출 가능한가" 이전에 "호출 권한이 있는가" 검증.

- **계정 발급 상태**:
  - dev/prod 계정 발급 완료?
  - 미발급 = R-XX (차단)
- **API key/인증 정보**:
  - 보유 여부 + 보관 위치(Secrets Manager 등)
  - dev/prod 분리?
- **테스트 환경 접근권한**:
  - dev endpoint 호출 가능?
  - 화이트리스트 IP 등록 필요?
- **영업 컨택 포인트**:
  - 담당자 이메일·전화
  - 미확보 = R-XX
- **계약 한도**:
  - Rate Limit, 월 한도, 단가
  - 운영 한도 초과 위험 시 R-XX
- **운영 전제 매트릭스 등록 의무 (P-51)**:
  - 설계서 9.X 운영 전제 매트릭스에 모든 외부 시스템 등록 확인
  - 매트릭스 ❓ 항목 = 자동 R-XX

### Step 3: 코드베이스 대조

기존 코드에서 유사 엣지케이스 대응 패턴 확인:
- 기존 SQS 리스너의 예외 처리 패턴 (AlimTalkAssembleNormalListener)
- 기존 배치의 중복 방지 패턴 (BatchNotification)
- 기존 차감 로직의 락 패턴 (ShopServiceImpl)

### Step 4: 대응 방안 제안 + Decision Gate 검증

각 엣지케이스에 대해:
1. **현재 설계에 반영 여부** 확인
2. **기존 코드에 유사 패턴 여부** (있으면 참조 경로 명시)
3. **대응 방안** 제안 (반영 안 된 경우)
4. **심각도** 분류 (Critical / Major / Minor)

#### Decision Gate 검증 (v2 신설)

설계서의 **R-XX 잔여 리스크**가 `decision-gate.md` 표준 포맷을 충족하는지 검증:

```
모든 R-XX에 대해:
- [ ] 차단 여부 명시?     (누락 = REJECT)
- [ ] 영향 범위 명시?
- [ ] 해결 액션 구체적?   ("리서치 필요"만 = REJECT)
- [ ] 담당자 명시?
- [ ] 기한 명시?
- [ ] 결정 문서 경로 (DECISION-XX.md)?
- [ ] 상태 (OPEN/IN_PROGRESS/RESOLVED/DEFERRED)?
```

**판정 룰 강화**:
- 차단 R-XX 미해결 1건 이상 → **LOOP 강제** (아키 재실행)
- R-XX 표준 포맷 미준수 → **LOOP 강제**
- 운영 전제 매트릭스 ❓ 항목 미등록 R-XX → **LOOP 강제**

**해결 액션 구체성 휴리스틱**:
- ❌ "리서치 필요", "확인 필요", "추후 결정"
- ✅ "올빼미 리서치 1시간 + DECISION-R08-1.md 작성"
- ✅ "영업 미팅 [담당] [기한] [질문 N개] + DECISION-R09.md"
- ✅ "파일럿 dev 환경 호출 + 응답 샘플 + DECISION-XX.md"
