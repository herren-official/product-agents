# 정책 검토 (Policy Review)

## 워크플로우

### Step 0: 학습 기록 로드

1. `.claude/agents/learnings/guardian_learnings.md` 파일 존재 확인 (Read 시도)
2. 존재하면:
   a. "활성 체크리스트" 섹션을 메모리에 로드
   b. 이후 점검 시 각 체크리스트 항목을 추가 검증 관점으로 활용
   c. 산출물의 "학습 패턴 체크 결과" 항목에 반영 여부 기록
3. 미존재 시: 건너뛰고 진행

### Step 1: 입력 자료 통합

3개 산출물을 읽고 대조 준비를 한다:
- 루나의 요구사항 → 기획 측 정책 목록
- 코드체커의 코드 매핑 → 코드 측 동작
- (있으면) 올빼미의 리서치 → 외부 시스템 제약

### Step 2: 정책 대조 (3방향)

각 기능에 대해 3방향 대조를 수행한다:

**A. 기획 → 코드 대조**
- 기획서에 명시된 비즈니스 룰이 코드에 구현되어 있는가?
- 구현이 기획 의도와 일치하는가?
- 불일치가 있다면 기획이 잘못인가, 코드가 잘못인가?

**B. 코드 → 기획 대조**
- 코드에 있지만 기획서에 언급되지 않은 로직이 있는가?
- 기존 코드의 validation/제약이 새 기획과 충돌하는가?

**C. 엣지케이스 점검**
- 다음 관점에서 양쪽 모두에 없는 케이스를 검출:
  - 동시성 (동시 요청, 중복 클릭)
  - 경계값 (0건, 최대건, 빈 값)
  - 상태 전이 (허용되지 않는 전이, 이미 처리된 건)
  - 권한 (다른 샵/직원/고객의 접근)
  - 시점 (만료, 기간 경과, 자정 경계)
  - 외부 장애 (타임아웃, 부분 실패)
  - 데이터 정합성 (관련 테이블 동시 갱신)

**D. 외부 의존 정책 검증 (v2 신설 — 2026-04-26)**

> 외부 API/벤더 사용 시 정책에 누락된 결정·전제가 있는지 검증.
> 도입 배경: 비즈뿌리오(인포뱅크 자회사) ≠ 톡드림(LG CNS 자체) 같은 벤더 혼동 사례, 그리고 "이미지 업로드 방식 미결정" 묵살 사례.

- **벤더 식별 정확성**:
  - 기획서가 벤더 이름·서비스명을 모호하게 적었는가?
  - 예: "LG CNS 사용" → 톡드림? 다른 자회사? 정확명 명시 필요
  - 코드베이스 endpoint(예: `talkapi.lgcns.com`)와 교차 확인
- **공개 API 부재 가능성 플래그**:
  - 기획서가 "API 사용"만 명시하고 가이드 출처가 없는가?
  - 폐쇄형 B2B 서비스(예: LG CNS 톡드림)는 영업 미팅 필수 → 기획 단계에서 미리 인지해야 함
- **운영 전제 누락**:
  - 기획서가 "외부 API 사용"이라고만 적고, **계정·API key·테스트 환경·영업 컨택**을 언급하지 않는가?
  - 운영 전제 미확정은 정책 누락으로 분류
- **결정 영역 누락**:
  - 외부 API 호출 흐름(사전 업로드 vs URL 전달 vs 인라인) 결정 여부?
  - 라이프사이클(TTL/만료/갱신) 결정 여부?
  - 동시성(race condition·분산 락) 결정 여부?
  - 미결정 = R-XX 등록 권고 (Decision Gate 적용)

### Step 3: 심각도 분류

검출된 항목을 심각도별로 분류한다:
- **Critical**: 데이터 정합성, 결제, 보안 관련
- **Major**: 비즈니스 로직 불일치, 상태 관리
- **Minor**: UX, 메시지, 표시 관련
- **Question**: 의도 불분명, 확인 필요

### Step 4: 확인 질문 작성

Woody님에게 확인할 질문을 작성한다.

**좋은 질문 형식:**
```
Q: {상황}에서 {케이스}가 발생하면 {어떻게 처리}하나요?
   코드에서는 {현재 동작}으로 되어 있고,
   기획서에서는 {기획 내용 또는 "언급 없음"}입니다.
```

**나쁜 질문 형식:**
```
Q: 이 부분 괜찮나요?
Q: 엣지케이스 처리가 필요할까요?
```

### Step 4.5: QA 루프 판단

루나와 코드체커 산출물의 품질을 평가하여 재실행 필요 여부를 판단한다.

**루나 산출물 평가:**
1. "정책 커버리지 자가 평가" 섹션에서 커버리지 확인 (< 70% → FAIL)
2. 기획서 raw_spec.md를 Read하여 주요 `##` 섹션 목록 추출
3. 루나가 식별한 정책과 대조 — 누락된 섹션이 있으면 FAIL
4. [불명확] 태그 없이 근거 불명확한 정책이 있으면 FAIL

**코드체커 산출물 평가:**
1. 루나의 기능 목록(F-01, F-02, ...) 전체 추출
2. 코드체커의 기능별 코드 매핑에서 각 F-XX 존재 확인 — 누락 2개 이상 FAIL
3. 핵심 enum/상수에 대한 영향 범위 주장을 3건 spot-check (Grep 재실행)
4. module_context.md와 SB 버전 일치 확인

**판단 결과를 "QA 루프 판단" 섹션에 기록한다.**

### Step 5: 산출물 작성

`_workspace/03_guardian_policy_gaps.md`에 에이전트 정의의 출력 형식대로 작성한다. QA 루프 판단 섹션을 반드시 포함한다.
