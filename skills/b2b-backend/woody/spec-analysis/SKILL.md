# 기획서 분석 (Spec Analysis)

## 워크플로우

### Step 0: 학습 기록 로드

1. `.claude/agents/learnings/luna_learnings.md` 파일 존재 확인 (Read 시도)
2. 존재하면:
   a. "활성 체크리스트" 섹션을 메모리에 로드
   b. 이후 분석 시 각 체크리스트 항목을 추가 점검 관점으로 활용
   c. 산출물의 "학습 패턴 체크 결과" 항목에 반영 여부 기록
3. 미존재 시: 건너뛰고 진행

### Step 1: 기획서 읽기

수집 스킬(notion-collector, figma-collector)이 생성한 파일을 읽는다.

1. **raw_spec.md (또는 raw_spec/ 디렉토리) 읽기**
   - 500줄 이상이면 Read 도구의 offset/limit으로 150줄씩 chunk 읽기
   - raw_spec/ 디렉토리면 index 파일을 먼저 읽고, 각 분할 파일을 순서대로 읽기
   - 반드시 파일 전체를 읽은 후 분석을 시작한다
2. **figma_context.md 읽기** (있는 경우)
   - 섹션 트리에서 화면 목록 추출
   - 핵심 화면별 요소 파악
3. 복수 소스 간 내용이 중복/충돌하면 소스 출처를 병기하고 `[충돌]` 태그 표시

### Step 2: 요구사항 구조화

기획서를 기능 단위로 분해한다.

**분해 기준:**
1. 사용자 행위 단위 (예: "예약 생성", "결제 취소")
2. 화면/API 단위로 나뉘어 있으면 그 구조를 따름
3. 하나의 기능이 여러 단계를 포함하면 단계별로 분리

**각 기능에서 추출할 항목:**
- 기능명 + 설명
- 입력 (사용자 입력, 시스템 입력)
- 출력 (화면 변화, 데이터 변경, 알림 등)
- 비즈니스 룰 (조건, 계산 로직, 유효성 검증)
- 예외/엣지케이스 (기획서에 명시된 것)
- 불명확 항목 (`[불명확]` 태그)

### Step 3: 정책/제약사항 식별

기획서 전체에서 횡단적 정책을 식별한다:
- 권한/접근 제어
- 상태 전이 규칙
- 데이터 유효성/제약
- 시간 관련 정책 (만료, 유예, 기한)
- 과금/결제 관련 정책

### Step 4: 외부 연동 감지

다음 키워드/맥락에서 외부 연동 필요성을 감지한다:
- 외부 API 호출 (네이버, 결제사, Shopby 등)
- 새로운 라이브러리/프레임워크
- 외부 시스템 데이터 동기화
- 법적/규제 요건

외부 연동이 감지되면 올빼미(researcher) 호출 필요 여부를 판단한다.

### Step 5: 디자인 교차 검증 (figma_context.md가 있는 경우)

1. 기획서의 기능 목록 vs 피그마 섹션 트리 대조
2. 기획에만 있는 기능 → `[디자인누락]` 태그
3. 디자인에만 있는 화면 → `[기획누락]` 태그
4. 결과를 산출물의 "디자인 교차 검증" 섹션에 기록

### Step 6: 산출물 작성

`_workspace/01_luna_requirements.md`에 에이전트 정의의 출력 형식대로 작성한다.
