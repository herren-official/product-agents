# TASK 분해 (Task Decomposition)

## 워크플로우

### Step 1: 설계 분석

1. `10_architect_design.md` 전체 읽기
2. 의존성 그래프에서 독립 노드(진입 간선 없음) 식별
3. 기능별 변경 파일 수 집계
4. `.docs/conventions/developer-guide/` 컨벤션 확인 (TASK 구현 시 따를 규칙)

### Step 2: TASK 분할 기준 적용

**마일스톤 순서 필수 준수:**
> 설계서의 마일스톤(M1→M2→...→MN) 순서가 곧 작업 우선순위다.
> 마일스톤을 재배열하지 않는다. 마일스톤 내에서만 TASK를 분할한다.

**마일스톤 내 분할 우선순위:**
1. 규격/Enum 정의 (common 모듈) → 항상 최우선
2. DB 마이그레이션 → 별도 TASK (2순위)
3. 독립 기능 → 기능별 TASK
4. 의존 기능 → 의존 순서대로 TASK 번호 부여

**TASK 크기 검증:**
- 변경 파일 10개 초과 → 서브 TASK로 분할
- 변경 파일 1~2개 → 인접 TASK와 병합 고려
- 예상 SP 3 초과 → 분할 검토

### Step 3: 의존 관계 정리

1. TASK 간 의존 관계를 DAG(방향성 비순환 그래프)로 정리
2. 순환 의존 발견 시: 인터페이스 분리 TASK를 선행으로 추가
3. 1인 순차 실행 기준, 마일스톤 순서 내에서 최적 순서 결정

### Step 4: TASK 골격 문서 작성

> **태스커는 골격만 작성한다.** 구현 코드, 테스트 코드, 상세 구현은 TASK 디테일러가 담당.

산출물 위치: `{작업디렉토리}/task/TASK-XX-키워드.md`

**골격 문서 양식:**

```markdown
# TASK-XX: 제목

## 개요
- **마일스톤**: MN (이름)
- **예상 SP**: N
- **의존**: 없음 (또는 TASK-XX)

## 브랜치
- **에픽 브랜치**: epic-{에픽명}
- **Base 브랜치**: epic-{에픽명} (또는 직전 TASK 브랜치)
- **작업 브랜치**: epic-{에픽명}-TASK-XX-키워드

## 목적
이 TASK가 해결하는 문제와 완료 시 기대 결과.

## 구현 범위
- 수정 모듈: {모듈명}
- 신규 파일 목록 (경로 + 클래스 책임)
- 기존 파일 변경 목록 (경로 + 변경 설명)

## 관련 정책
- 설계서 참조 섹션 (예: 7.1 차감/원복)
- 적용할 비즈니스 룰 요약

## 완료 조건
- [ ] ...
```

### Step 6: 요약 산출물 작성

`_workspace/11_tasker_tasks.md`에 전체 TASK 요약 작성:

**필수 포함:**
1. TASK 전체 목록 테이블 (번호, 제목, 마일스톤, SP, 의존, 모듈)
2. 의존 관계 다이어그램 (Mermaid)
3. 순차 실행 순서
4. 브랜치 전략 (에픽 브랜치 → TASK 브랜치 체인)
5. 개별 TASK 문서 링크

**브랜치 전략:**
```
develop
  └── epic-{에픽명} (에픽 브랜치, 최종 PR → develop)
       └── epic-{에픽명}-TASK-01-xxx
            └── epic-{에픽명}-TASK-02-xxx
                 └── ...
```
- TASK PR: 직전 TASK 브랜치 → epic 브랜치
- 최종 PR: epic 브랜치 → develop
