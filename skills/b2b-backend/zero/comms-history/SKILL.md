특정 주제(에픽, 기능, 이슈 등)에 대한 슬랙/노션 커뮤니케이션 이력을 수집하여 시간순 정리 문서를 생성한다.

## Phase 0: 인자 처리 & 사용자 확인

`$ARGUMENTS`가 비어있으면 사용자에게 주제를 물어본다.

`$ARGUMENTS`가 제공된 경우에도 **반드시** 사용자에게 아래 항목을 질문한다:

1. **검색 범위**: 특정 슬랙 채널/스레드가 있는지? 노션 페이지 링크가 있는지?
2. **기간**: 언제부터의 이력을 수집할지 (예: 킥오프부터, 특정 날짜부터)
3. **이미 알고 있는 링크**: 에픽, 피그마, 회의록 등 이미 파악된 문서 링크가 있으면 공유해달라

이 정보를 받은 후 Phase 1을 시작한다. 사용자가 "알아서 찾아줘"라고 하면 키워드 기반으로 검색을 시작한다.

## Phase 1: 소스 수집 (병렬 에이전트)

3개 에이전트를 **동시에** 실행하여 관련 자료를 수집한다.

### Agent 1: slack-searcher

```
subagent_type: general-purpose
name: slack-searcher
```

슬랙에서 관련 스레드와 메시지를 검색한다.

작업 내용:
1. Slack MCP의 `slack_search_public_and_private`로 주제 키워드 검색
2. 검색 결과에서 관련 채널/스레드 식별
3. `slack_read_thread`로 주요 스레드 전문 읽기
4. 각 메시지에서 추출:
   - **날짜/시간**
   - **발화자** (닉네임)
   - **내용 요약** (결정사항, 질문, 답변, 공유 등 성격 구분)
   - **스레드 링크**
5. 결과를 날짜순으로 정리하여 반환

### Agent 2: notion-searcher

```
subagent_type: general-purpose
name: notion-searcher
```

노션에서 관련 문서를 검색한다.

작업 내용:
1. Notion MCP의 `notion-search`로 주제 키워드 검색
2. 관련 페이지(에픽, 회의록, 설계문서, API 문서 등) 식별
3. `notion-fetch`로 주요 문서 내용 읽기
4. 각 문서에서 추출:
   - **문서 유형** (에픽, 회의록, 설계문서, API 문서 등)
   - **날짜** (작성일/회의일)
   - **핵심 내용** (결정사항, 논의사항, 스펙 등)
   - **노션 URL**
5. 결과를 날짜순으로 정리하여 반환

### Agent 3: local-searcher

```
subagent_type: Explore
name: local-searcher
```

로컬 리소스에서 관련 파일을 검색한다.

작업 내용:
1. `~/claude-resources/` 하위에서 관련 키워드 검색 (Grep)
2. 메모리 파일(`~/.claude/projects/-Users-herren/memory/`)에서 관련 내용 확인
3. 이미 정리된 문서가 있으면 해당 내용 반환 (중복 작업 방지)
4. 피그마 링크 등 외부 참조 URL 수집

## Phase 2: 이력 통합 & 검증

Phase 1 결과를 모두 받은 후 통합 작업을 수행한다.

### 통합 규칙

1. **시간 역순 정렬**: 모든 이벤트를 날짜별로 그룹핑, 최신이 위로
2. **요일 검증**: `cal` 명령어로 해당 월 달력을 확인하여 요일 정확히 표기
3. **사실 구분 — 가장 중요**:
   - **확정**: 의사결정권자가 명시적으로 "확정", "이걸로 가겠습니다" 등 결정한 것만
   - **공유/논의**: 미팅에서 공유된 추정치, 예상일정, 방향성 논의
   - **요청/질문**: 누군가 요청하거나 질문한 것
   - **미답변**: 질문이 나왔지만 답변이 없는 것
   - 킥오프에서 나온 공수 추정, 예상일정은 "확정"이 아니라 "공유"로 표기
4. **링크 필수**: 모든 항목에 출처 슬랙 스레드 또는 노션 문서 링크 포함. 괄호 안 텍스트 참조도 클릭 가능한 링크로.
5. **노이즈 제거**: 인지해야 할 핵심 내용만. 표 형식 금지, 불릿 리스트로 간결하게.

### 교차 검증

슬랙 메시지와 노션 회의록을 교차 확인하여:
- 회의록에 "확정"으로 적힌 것이 실제 확정인지 맥락 확인
- 시간 흐름상 뒤에서 뒤집힌 결정이 있는지 확인
- 미확정 상태로 남은 것이 이후 확정되었는지 추적

## Phase 3: 문서 생성

`~/comms-history/` 에 마크다운 파일로 저장한다. 디렉토리가 없으면 생성한다.

### 문서 구조

```markdown
# {분기_프로젝트명} (예: 26-1Q 마케팅 수신 동의FCM)

## {주제} — 전체 커뮤니케이션 이력

> 최종 업데이트: {오늘 날짜} ({요일})

---

## 1. 에픽 문서

- **에픽**: [에픽명](url)
- **피그마**: [피그마명](url)  (있는 경우)
- 기타 주요 문서 링크

**에픽 문서 내용:**
1. (에픽에서 확정된 스펙 요약)

---

## 2. 커뮤니케이션 이력 (시간 역순)

### MM/DD (요일) — 한줄 요약

하루에 소스가 하나면 날짜 헤더 아래 바로 링크:
[출처 링크](url)

하루에 여러 토픽이면 소제목으로 분리:
**[토픽명]** - [출처 링크](url)
**[토픽명]** ([출처](url))

- 누구: 무엇
- 누구: 무엇

---

## 3. 주요 결정사항 (시간순)

확정된 의사결정만 날짜순으로 나열. 각 항목에 출처 링크 포함.
형식: `**MM/DD** 결정 내용 — 누가/출처 ([링크](url))`

---

## 4. 주요 결정 흐름 (선택)

특정 이슈가 여러 날에 걸쳐 결정된 경우, 별도 섹션으로 의사결정 타임라인 정리.
각 항목에 출처 링크 포함.

---

## 5. 미확정/추가 논의 필요

- 항목 — 설명 ([출처](url))

---

## 6. 관련 문서/스레드 링크

지속적으로 인지가 되어야할 것 같은 문서/링크를 정리합니다.

| 구분 | URL |
|------|-----|
| ... | ... |
```

### 파일명 규칙

- 주제를 kebab-case 영문으로: `{topic}-comms-history.md`
- 예: `marketing-consent-history.md`, `sms-refactoring-history.md`

## Phase 4: 사용자 검토

문서 생성 후:
1. 전체 내용을 마크다운 텍스트로 출력 (사용자가 직접 확인 가능하도록)
2. "확정"으로 표기한 항목 목록을 별도로 보여주며 정확한지 확인 요청
3. 미확정 항목 목록도 보여주며 누락 없는지 확인
4. 사용자 피드백 반영하여 수정
