# 자동 커밋 생성

현재 변경사항을 분석하여 프로젝트 커밋 룰에 따라 자동으로 커밋을 생성합니다.

## 사용법
```bash
/create-commit              # 기본 자동 커밋
/create-commit "추가 요구사항"  # 추가 컨텍스트와 함께 커밋
```

## 처리 단계

1. **변경사항 분석**
   - `git status`로 현재 변경 파일 확인
   - `git diff`로 실제 변경 내용 검토
   - 파일 타입별 그룹화 (문서/테스트/컴포넌트/설정 등)

2. **노션 ID 추출**
   - 현재 브랜치명에서 GBIZ-XXXXX 패턴 추출
   - 노션 ID 발견 시 scope로 사용
   - 브랜치명 예: `GBIZ-19723-feature-description`

3. **커밋 전략 결정**
   - 변경 파일 수와 타입 분석
   - [Git 커밋 컨벤션](/.docs/conventions/COMMIT_CONVENTION.md)에 따라:
     - 단일 작업: 하나의 커밋으로 처리
     - 복합 작업: **작업 유형별로 반드시 커밋 분리**
   - 커밋 분리 원칙:
     - **서로 다른 타입의 변경사항은 절대 같이 커밋하지 않음**
     - 예: 문서(docs)와 테스트(test)는 별도 커밋
     - 예: 기능(feat)과 리팩토링(refactor)은 별도 커밋
     - **같은 파일이라도 작업 내용이 다르면 별도로 커밋**
   - 커밋 계획 자동 결정

4. **커밋 메시지 작성**
   - 커밋 타입 자동 결정:
     - `feat`: 새로운 기능 추가
     - `fix`: 버그 수정
     - `docs`: 문서 수정
     - `style`: 스타일 코드 수정
     - `refactor`: 코드 리팩토링
     - `test`: 테스트 코드 추가 또는 수정
     - `chore`: 빌드 작업, 패키지 매니저 설정 등
   
   - 형식 (⚠️ 본문과 꼬리말 없이 제목만 작성):
     ```
     type(GBIZ-XXXXX): 간결한 한글 설명
     ```
   
   - 규칙:
     - 한글로 작성
     - 현재형 동사로 시작
     - 마침표(.) 없음
     - 50자 이내
     - **"및", "와/과" 등 접속사 사용 금지** (커밋 분리의 신호)

5. **커밋 실행 전 검증**
   - **[Git 커밋 컨벤션](/.docs/conventions/COMMIT_CONVENTION.md) 전체 내용 재확인**
   - 주요 검증 항목:
     - 커밋 단위 분리 원칙 (132-176번 라인)
     - 필수 규칙 준수 여부 (101-112번 라인)
     - 작업 단위 분리 가이드 (145-156번 라인)
     - 피해야 할 사항 (121-127번 라인)
   - 검증 실패 시 커밋 중단하고 재분석

6. **커밋 실행**
   - 검증 통과한 파일만 `git add` 실행
   - 준비된 메시지로 `git commit` 실행
   - 각 커밋 후 상태 확인

7. **결과 보고**
   - 생성된 커밋 목록 표시 (`git log --oneline -n`)
   - 최종 상태 확인 (`git status`)

## 추가 컨텍스트 처리

사용자가 추가 요구사항을 제공한 경우:
- 커밋 메시지에 추가 정보 반영 (제목 한 줄에만)
- 특정 파일 우선순위 조정
- 커밋 분리 전략 커스터마이징

## 예시

### 기본 사용
```bash
/create-commit
# 자동으로 변경사항 분석 후 커밋 생성
```

### 추가 컨텍스트와 함께
```bash
/create-commit "성능 최적화 중점으로 커밋 메시지 작성"
/create-commit "모든 테스트 파일은 별도 커밋으로"
/create-commit "API 관련 변경사항 강조"
```

### 자동 커밋 분리 예시

변경된 파일:
- `docs/README.md` (문서 수정)
- `src/components/Button.test.tsx` (테스트 추가)
- `src/components/Button.tsx` (기능 추가)

자동으로 3개의 커밋으로 분리:
1. `docs(GBIZ-1234): README에 버튼 컴포넌트 사용법 추가`
2. `test(GBIZ-1234): Button 컴포넌트 단위 테스트 추가`
3. `feat(GBIZ-1234): Button 컴포넌트에 disabled 상태 추가`

## 커밋 메시지 예시

```bash
feat(GBIZ-1234): 고객 상세 페이지에 메모 기능 추가
fix(GBIZ-1234): 등록된 매출 > 결제수단 변경 시 NPE 수정
test(GBIZ-1234): Button 컴포넌트 단위 테스트 추가
refactor(GBIZ-1234): 사용하지 않는 import 제거
style(GBIZ-19249): 날짜 선택 버튼 셀렉트와 동일한 스타일 적용
```

## 주의사항

- **본문(body)과 꼬리말(footer) 절대 작성 금지**
- **서로 다른 타입의 변경사항은 절대 하나의 커밋으로 만들지 않음**
  - 잘못된 예: "test(GBIZ-1234): 테스트 추가 및 문서 업데이트" ❌
  - 올바른 예: 테스트와 문서를 별도 커밋으로 분리 ✅
- **커밋 메시지에 "및", "와", "과" 등의 접속사가 들어가면 커밋 분리 필요**
- 빌드가 깨지지 않도록 의존성 있는 파일은 함께 커밋
- 너무 세밀한 분리 지양 (가독성 우선)
- PR 리뷰어 관점에서 논리적 그룹화
- 머지 커밋과 릴리즈 커밋은 수동 처리
