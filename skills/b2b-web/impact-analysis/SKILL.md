# Impact Analysis

기능 활성화, 변경, 제거 시 **서비스 전반에 미치는 정책적·비즈니스적 영향도**를 분석하는 스킬.

## 명령어

```
/impact-analysis {기능 설명}
/impact-analysis 고객차트 필터 중 담당자, 받은 시술 활성화
/impact-analysis 정액권 환불 기능 추가
/impact-analysis --deep {기능 설명}    # 심층 분석 (데이터 정합성, 엣지케이스까지)
```

## 차별점

| 스킬 | 초점 |
|------|------|
| `kickoff-edge-scan` | 개발 착수 전 Figma/코드의 **엣지케이스·정책 공백** 탐지 |
| `verify-policy-coverage` | 정책-코드-테스트 **3중 매핑 커버리지** 검증 |
| **이 스킬** | 기능 변경이 **서비스 전반에 미치는 연쇄 영향** 분석 (정책·비즈니스·데이터·운영 관점) |

## 팀 기반 병렬 실행 아키텍처

> 전제: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` + `teammateMode: "tmux"` 설정 필요

### 팀 구조

```
┌─ tmux %0: team-lead (오케스트레이터)
│   └─ Phase 3(교차 분석) + Phase 4(리포트) 직접 수행
│
├─ tmux %1: policy-analyst (Tier 1)
│   └─ Phase 1: 옵시디언 정책 영향도 분석
│
└─ tmux %2: code-analyst (Tier 1)
    └─ Phase 2: 코드베이스 영향도 분석
```

### 의존성 그래프

```
                 ┌─ policy-analyst ────┐
Tier 1 (병렬)   │  Phase 1            │
                 │  → impact-policy.md │
                 │                     ├─→ Leader: Phase 3 (교차 분석)
                 │                     │   → impact-cross.md
                 └─ code-analyst ──────┤
                    Phase 2            │
                    → impact-code.md   │
                                       └─→ Leader: Phase 4 (리포트 병합)
                                           → {feature}-impact-analysis.md
```

### 실행 흐름

```
1. TeamCreate("{feature}-impact")
2. TaskCreate x 2:
   - Task 1: "정책 영향도 분석 — 옵시디언 정책 문서 탐색"
   - Task 2: "코드 영향도 분석 — 코드베이스 연쇄 영향 탐색"
3. Agent x 2 (동시 spawn, tmux 분할):
   - policy-analyst → Task 1 수행
   - code-analyst → Task 2 수행
4. 각 teammate: TaskUpdate(completed) → idle notification
5. Leader: idle 수신 → Phase 3 + Phase 4 직접 수행
6. Shutdown protocol:
   - SendMessage(shutdown_request) to each teammate
   - Wait for shutdown_response(approve)
   - TeamDelete
```

## 임무

## 절차

### Step 1: 직접 관련 정책 탐색
1. policies/ 디렉토리 전체 구조 파악
2. 기능과 직접 관련된 정책 파일 모두 읽기
3. screens/ 하위의 관련 화면 정책 읽기

### Step 2: 도메인 교차 정책 탐색
기능이 **다른 도메인과 만나는 지점**을 찾기:
- 예약 ↔ 시술 ↔ 매출 ↔ 정액권/티켓 ↔ 고객 관계
- 알림 ↔ 고객 ↔ 메시지 관계
- 권한 ↔ 기능 접근 관계
- 통계 ↔ 데이터 소스 관계

### Step 3: 8가지 정책 영향도 체크리스트

| # | 체크포인트 | 분석 방법 |
|---|-----------|----------|
| 1 | 개념 정의 | 이 기능의 핵심 개념이 정책적으로 어떻게 정의되는지 (예: "받은 시술" = 매출 등록된 시술) |
| 2 | 데이터 흐름 | 이 기능의 데이터가 어디서 생성되고, 어디서 소비되는지 (생성→가공→표시→저장) |
| 3 | 도메인 간 의존성 | 다른 도메인(예약/매출/시술/고객/알림)과의 정책적 연결점 |
| 4 | 상태 전이 영향 | 기능 활성화가 기존 상태 전이 다이어그램에 미치는 변화 |
| 5 | 권한/역할 영향 | 기능 사용에 필요한 권한, 역할별 접근 차이 |
| 6 | 사용자 시나리오 | 매장에서 이 기능을 실제로 어떻게 사용할지 (핵심 유스케이스 3~5개) |
| 7 | 운영 리스크 | 데이터 변경/삭제 시 이 기능에 미치는 영향 (시술 삭제, 직원 퇴사 등) |
| 8 | 미해결 정책 | 정책 문서에 명시되지 않은 모호한 지점 (서버 확인 필요 사항) |

## 임무
코드베이스를 탐색하여 기능 변경의 기술적 영향 범위를 분석합니다.
프론트엔드 수정 범위가 아니라, **서비스 관점의 기술적 영향도**에 집중합니다.

## 입력
- 기능 설명: {featureDescription}
- Feature name: {featureName}
- 도메인 키워드: {domainKeywords}

## 절차

### Step 1: 영향 범위 매핑
1. 기능과 직접 관련된 컴포넌트/훅/페이지 전수 조사
2. 관련 API 엔드포인트와 data-contracts.ts 타입 확인
3. Recoil atom/selector 의존성 추적
4. 관련 queryFactory 전수 조사

### Step 2: 간접 영향 추적
기능이 **다른 기능과 공유하는 코드**를 추적:
- 같은 API를 사용하는 다른 기능
- 같은 Recoil atom을 읽는 다른 컴포넌트
- 같은 queryFactory를 사용하는 다른 훅
- 같은 타입을 사용하는 다른 모듈

### Step 3: 10가지 서비스 영향도 체크리스트

| # | 체크포인트 | 분석 방법 |
|---|-----------|----------|
| 1 | 기능 사용처 전수 조사 | 이 기능이 사용되는 모든 페이지/컴포넌트 (직접+간접) |
| 2 | 데이터 소스 불일치 | 같은 개념인데 다른 API/소스를 사용하는 곳 |
| 3 | API 타입 완성도 | 요청/응답 타입에 필요한 필드가 모두 있는지 |
| 4 | 상태 공유 범위 | Recoil atom이 예상보다 넓게 공유되는지 |
| 5 | 페이지 간 데이터 전달 | sessionStorage, query param 등으로 다른 페이지에 영향 |
| 6 | 권한 코드 매핑 | 기능에 필요한 권한 코드와 실제 체크 로직 |
| 7 | 캐시 무효화 | 기능 변경 시 어떤 쿼리 캐시가 무효화되어야 하는지 |
| 8 | 사이드이펙트 | 기능 활성화가 기존 동작에 영향을 줄 수 있는 코드 경로 |
| 9 | 데이터 정합성 | 삭제/수정 시 참조 무결성이 깨질 수 있는 지점 |
| 10 | 성능 영향 | 추가 API 호출, 대량 데이터 처리, 리렌더링 범위 |

## Phase 3: 교차 분석 (Leader 직접 수행)

> **두 teammate 완료 후** leader가 `impact-policy.md` + `impact-code.md`를 읽고 수행.

### 교차 분석 포인트

| # | 분석 | 방법 |
|---|------|------|
| 1 | 정책 vs 코드 갭 | 정책에는 정의되어 있지만 코드에 구현 안 된 것, 또는 그 반대 |
| 2 | 데이터 소스 불일치 심화 | 정책의 개념 정의와 코드의 실제 데이터 소스가 다른 곳 |
| 3 | 연쇄 영향 시나리오 | "A가 바뀌면 → B에 영향 → C도 영향"의 체인 도출 |
| 4 | 운영 리스크 + 코드 리스크 교차 | 운영 시나리오(삭제/변경)가 코드에서 어떻게 처리되는지 |
| 5 | 서버 확인 필요 사항 통합 | 정책 미해결 + 코드 미확인을 통합하여 우선순위 부여 |

## Claude 행동 규칙

1. **서비스 관점 우선** — 프론트엔드 코드 수정 범위가 아닌, 비즈니스/정책/운영 영향에 집중
2. **"영향 없음"도 보고** — 영향이 없는 영역을 명시적으로 확인하여 누락 없음을 보장
3. **연쇄 영향 추적** — 1차 영향뿐 아니라 2차, 3차 연쇄 영향까지 추적
4. **증거 기반** — 모든 영향도 판단에 정책 문서 ID 또는 코드 file:line 근거 첨부
5. **운영 시나리오 중심** — 개발자가 아닌 매장 운영자 관점에서 영향을 서술
6. **미확인 사항 명시** — 서버 동작, 기획 의도 등 확인이 필요한 사항을 별도 섹션으로 분리
7. **팀 라이프사이클 준수** — TeamCreate → TaskCreate → Agent spawn → idle 대기 → Shutdown → TeamDelete
